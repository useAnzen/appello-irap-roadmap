EVAL-1 — Model Evaluation on Appello Codebase
Sprint 1 · Apr 2026 ~2 weeks Team: Corey
Systematic evaluation of 4 frontier AI models against real Appello code to select the best fit for each agent role. Not a generic benchmark — every test used actual Appello TypeScript/React code from the production monorepo.
What Was Built
  • Evaluation framework — Built a repeatable benchmark harness that takes an Appello code sample + prompt and runs it against each model, scoring output on 4 dimensions: code comprehension, fix accuracy, style compliance, and cost per query.
  • 50 real code samples curated — Selected 50 TypeScript/React code samples from the Appello monorepo representing real production complexity: Prisma schema operations, GraphQL resolvers, React components with Tailwind, Next.js API routes, and financial calculation modules.
  • 4 models evaluated — GPT-4, Claude 3.5 Sonnet, Gemini 1.5 Pro, and Llama 3 70B. Each ran against all 50 samples with identical prompts. Results stored in structured format for comparison.
  • Model routing strategy defined — Based on results: Claude 3 Haiku for simple parsing/classification tasks (fast, cheap), GPT-4o/Claude Sonnet for complex reasoning and code generation (highest accuracy). Cost optimization via routing reduces average cost per agent call by ~60% vs. using frontier model for everything.
Outcomes Delivered
  • All 4 models scored on 50 Appello-specific code samples across 4 dimensions
  • Primary model selected for code generation (GPT-4o / Claude Sonnet depending on task type)
  • Cost model established: ~$0.08 per simple query (Haiku), ~$2.50 per complex generation (Sonnet/GPT-4o)
  • Model routing strategy documented — Haiku for parsing, Sonnet/GPT-4o for reasoning
  • Benchmark harness reusable for future model evaluations (new models, fine-tuned versions)
Sample Benchmark Result
Financial Summary Component — Travel Expense Roll-up Bug

BRA-1 — Bug Report Agent v1
Sprints 2–4 · Apr–Jun 2026 ~6 weeks Depends: EVAL-1 Team: Corey (NH1 deferred to June)
The first AI agent in the SDLC Flywheel. Ingests a Jira bug ticket, parses the Appello monorepo (TypeScript/React/Express/Prisma), identifies the root cause, and generates a candidate fix diff as a GitHub PR. Core architecture built by Corey solo; full 20-bug validation deferred to R2 due to NH1 hire delay.
What Was Built
  • Jira ticket ingestion pipeline — Reads ticket via Jira API, extracts summary, description, reproduction steps, affected modules, and linked issues. Normalizes into structured format for agent consumption.
  • Codebase parser — Parses the Appello monorepo structure: identifies affected packages (Turbo Repo workspace), resolves import chains, and builds a local dependency graph for the flagged module. Handles TypeScript, React JSX, Prisma schema, and GraphQL resolvers.
  • Root cause analysis engine — LLM-powered analysis that takes the parsed ticket + relevant code context and generates a hypothesis about the root cause. Uses model routing (Haiku for initial classification, Sonnet for deep analysis).
  • Fix diff generator — Generates a candidate code fix as a clean git diff, targeting the specific files identified by the root cause analysis. Validates generated code against TypeScript compiler before outputting.
  • PR creation pipeline — Auto-creates a GitHub PR from the fix diff with structured description: linked Jira ticket, root cause summary, change description, and files modified. Follows Appello PR conventions.
Status & What Carries to R2
  • Core architecture: COMPLETE — End-to-end pipeline from Jira ticket → codebase parse → root cause → fix diff → PR is operational. Built by Corey as sole developer during R1.
  • 20-bug validation: DEFERRED to R2 — Full validation against 20 real customer bug reports requires NH1 (ML Engineer) for systematic testing, result analysis, and agent tuning. NH1 start delayed from May to June; validation will be first priority when NH1 + NH2 onboard in July.
  • Agent tuning & accuracy iteration: DEFERRED to R2 — Based on validation results, iterate on prompt engineering, context window management, and model routing to achieve target accuracy (≥60% valid fix rate on first attempt).
Architecture
Runs on the Catalyst/Agent C2 orchestration platform. The agent operates in a sandboxed environment per ticket to prevent state leakage between runs. Model routing: Haiku for ticket parsing and code classification, Sonnet/GPT-4o for root cause analysis and code generation.
Data Flow
Jira Ticket (API) → Ticket Parser (Haiku) → Codebase Parser (AST) → Root Cause Analyzer (Sonnet) → Fix Generator (GPT-4o) → tsc Validation → GitHub PR (API)

SDLC-DB — Pipeline Tracking Schema
Sprints 3–4 · May–Jun 2026 ~4 weeks Team: Chris, Travis
Database schema and API layer for tracking every step of the SDLC pipeline. Every agent run, every task decomposition, every code review, and every deployment is recorded here — this is the “memory” that powers the Flywheel’s self-improvement loop.
What Was Built
  • 8 database tables — tasks (Jira ticket mirror), signals (bug reports, feature requests), decompositions (PA-1 subtask plans), reviews (VA-1 quality scores), deployments (release tracking), metrics (agent performance over time), agents (agent registry and config), history (full audit trail of every state change).
  • 12 REST API endpoints — CRUD operations for all entities plus specialized query endpoints: tasks by status, reviews by quality score range, metrics aggregations by time period, and a full pipeline trace endpoint (ticket → plan → code → review → deploy).
  • Migration framework — Prisma-based schema management with versioned migrations and tested rollback procedures. Each migration includes both up and down scripts.
  • Performance indexing — Composite indexes on frequently queried columns (task status + created date, review score + agent ID). Query performance validated under simulated load of 10,000 pipeline runs.
Outcomes Delivered
  • 8-table schema deployed to production with Prisma migrations
  • 12 REST API endpoints operational with authentication and rate limiting
  • Full audit trail: every agent action is logged with timestamp, input, output, and duration
  • Pipeline trace query returns complete ticket-to-deploy history in <200ms
  • Migration framework tested with 3 schema iterations during R1 development
Schema Overview
tasks → signals → decompositions → reviews → deployments · Cross-cutting: metrics, agents, history

ATLAS-1 — Data Pipeline Scaffold
Sprints 2–4 · Apr–Jun 2026 ~4 weeks Team: Chris
Foundation of the ATLAS knowledge base — the centralized AI memory that all agents query for context. Ingests company documents and workflow data into a vector store, enabling AI agents to ground their responses in real Appello business context.
What Was Built
  • Supabase pgvector store configured — PostgreSQL database with the pgvector extension for 1536-dimensional embeddings. Configured for cosine similarity search with HNSW indexing for sub-100ms query latency at scale.
  • Google Drive ingestion pipeline — Python scripts that crawl the Appello Google Drive, extract text from documents (meeting notes, architecture decisions, product specs, customer communications), chunk into 1000-token segments, generate embeddings via OpenAI API, and upsert into Supabase.
  • N8n workflow data pipeline — Ingestion connector for N8n automation workflow data (execution logs, workflow definitions, error reports). Converts structured workflow data into natural language descriptions for vectorization.
  • Fathom transcript ingestion — Processes meeting transcripts from Fathom (customer calls, internal standups, strategy sessions). Extracts action items, decisions, and key discussion points as separate vector entries for high-relevance retrieval.
  • Chunking and embedding strategy — 1000-token chunks with 200-token overlap to preserve context across boundaries. Metadata tags for source type, date, participants, and topic classification. Hybrid search combining vector similarity + metadata filtering.
Outcomes Delivered
  • Supabase pgvector store operational with HNSW index for fast similarity search
  • Google Drive documents ingested and vectorized (meeting notes, specs, architecture docs)
  • N8n workflow execution data flowing into ATLAS
  • Fathom meeting transcripts indexed with action item extraction
  • Query latency <100ms for similarity search across all ingested data
  • Foundation ready for ATLAS-2 (MySQL schema vectorization in R2)
Data Flow
Google Drive / N8n / Fathom → Python Ingestion Scripts → Text Extraction + Chunking (1000 tokens) → OpenAI Embeddings API → Supabase pgvector (HNSW index) → Agent Context Queries

FV-1 — Construction Training Data Collection
Sprints 1–4 · Apr–Jun 2026 ~8 weeks (part-time) Team: Nathan, Ian
AI models need domain-specific context to be useful for construction management software. This work stream collected, categorized, and formatted real customer data and industry terminology so the AI agents can understand Appello’s domain.
What Was Built
  • 30 real customer bug reports collected — Curated from Jira backlog: timesheet calculation errors, financial summary display bugs, equipment form validation issues, RFI workflow problems, and job costing discrepancies. Each annotated with root cause, affected module, fix description, and severity.
  • Bug report categorization schema — Classified bugs into 6 categories: financial calculations (8), UI/display (7), data integrity (5), workflow/routing (4), permissions (3), performance (3). This distribution informs which modules the AI agents need to handle best.
  • 40+ construction-specific terms documented — Domain glossary covering Appello-specific terminology: job costing, progress billing, change orders, RFIs, submittals, punch lists, retainage, WBS codes, cost codes, time & materials, lump sum, certified payroll, prevailing wage. Each term includes definition, how it appears in Appello, and common customer scenarios.
  • 15 construction job scenarios written — End-to-end scenarios representing real customer workflows: “Residential renovation project from estimate to final invoice,” “Commercial subcontractor managing certified payroll across 3 job sites,” “GC tracking change orders and budget impact on a multi-phase project.” Used to validate that AI agents produce contextually appropriate responses.
  • Customer signal analysis — Reviewed HubSpot support tickets and Fathom call transcripts to identify the most common customer pain points and feature requests. Prioritized 12 high-frequency issue patterns for AI agent training data.
Outcomes Delivered
  • 30 annotated bug reports ready as AI agent training and validation data
  • 6-category bug taxonomy informing agent specialization priorities
  • 40+ construction term glossary integrated into ATLAS for agent context
  • 15 end-to-end construction job scenarios for agent validation testing
  • 12 high-priority customer pain points identified for AI agent targeting
