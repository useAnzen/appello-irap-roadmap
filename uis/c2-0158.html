1 · Objective

agentc2/apps/agent/src/lib/inngest.ts is the queue/orchestration layer (with retry + step semantics); agentc2/apps/agent/src/lib/redis.ts is wired up; agentc2/packages/agentc2/src/tools/run-management-tools.ts manages queued runs. This epic extends Inngest with custom priority + retry policies for Spider test jobs: P0 customer-blocking flows preempt P2 regression sweeps; failed jobs retry with exponential backoff; saturated queue elevates older jobs.

2 · Acceptance Criteria

  • Inngest function spider/run-test accepts { priority: 'P0'|'P1'|'P2', flow, target, hamburgerSlug? }.
  • Priority enforcement: P0 jobs evicted-to-front via Redis sorted-set ordering; in-flight P2 paused if a P0 arrives and concurrency cap is hit.
  • Retry policy: 3 attempts with exponential backoff (1m/5m/30m); after exhaustion, write to SpiderFailure table for triage.
  • Aging: every 15 min, jobs older than P0=15m / P1=2h / P2=24h get priority bump.
  • Concurrency cap: 10 jobs total; per-priority sub-caps (P0 unlimited within total, P1 max 6, P2 max 3).
  • Observability: queue-depth + age histograms on the C1-0005 dashboard; alerts when P0 age > 5m.
  • End-to-end test: enqueue 20 P2 + 1 P0; assert P0 starts within 30s.

3 · Architecture & Approach

Priority queue model

// Redis sorted-set: spider:queue
// score = priority_weight*1e12 + ageMs (lower score = higher priority)
const score = priorityWeight[priority] * 1e12 + Date.now();
await redis.zadd('spider:queue', score, JSON.stringify(job));

Inngest function with priority

inngest.createFunction(
  { id: 'spider/run-test', concurrency: { limit: 10 } },
  { event: 'spider/job.enqueued' },
  async ({ event, step, attempt }) => {
    if (attempt > 0) await step.sleep(`backoff-${attempt}`, [`1m`,`5m`,`30m`][attempt-1]);
    try {
      return await runPlaywright(event.data);
    } catch (e) {
      if (attempt >= 2) await db.spiderFailure.create({ data: { ... } });
      throw e;  // Inngest retry
    }
  }
);

5 · Risks & Mitigations

  • Inngest concurrency semantics not perfectly fitting priority eviction. Mitigation: dispatcher pre-filters jobs before invoking; Inngest just handles the post-dispatch run.
  • Aging causes priority inversion confusion. Mitigation: aged jobs visibly tagged 'aged-P0' in dashboard.
  • Failed-job table unbounded. Mitigation: 30d retention + summary rollup.
