1 · Objective

agentc2/packages/agentc2/src/mcp/api-clients/jira.ts is the existing Jira integration; appello-sdlc work_packages_v2 + work_items + appello-roadmap/plans/ are the tracking surfaces today. This epic sets up the Jira project structure (projects, components, issue types, custom fields) + the dashboards that connect Jira back to work_packages_v2 / plans/ so external stakeholders see one coherent view and engineering operates from the unified backlog (C1-0038).

2 · Acceptance Criteria

  • Jira project APP structured with components matching appello-roadmap phases (P1-P5) + epics matching C1-C4.
  • Custom fields: workPackageSlug (link to work_packages_v2.slug), hamburgerSlug (link to work_items.slug), trustScore (mirrored from Review).
  • Webhook from Jira → appello-sdlc inngest function syncs Jira status changes into work_packages_v2.externalStatus.
  • Reverse sync: when work_packages_v2.state transitions, Jira issue auto-transitioned via MCP.
  • Dashboards: Current sprint, Aging epics, Trust-score distribution, Customer-impact-tagged backlog.
  • Stakeholder access: read-only Confluence-embed of the dashboards.
  • Documentation at appello-roadmap/processes/jira-conventions.md.

3 · Architecture & Approach

Sync flow

Jira issue updated
  → webhook to inngest /jira-webhook
  → match by customField workPackageSlug
  → update work_packages_v2.externalStatus, externalAssignee

work_packages_v2.state change (DB trigger)
  → inngest function sync-to-jira
  → jira.ts MCP transition(issueId, targetStatus)

4 · Implementation Phases (20h)

  1. Jira project + issue-type + field config (4h).
  2. Webhook handler in inngest (4h).
  3. Reverse sync via DB triggers (4h).
  4. 4 dashboards (4h).
  5. Confluence embed + access (2h).
  6. Conventions doc (2h).

5 · Risks & Mitigations

  • Status taxonomy mismatch Jira vs DB. Mitigation: mapping table maintained in jira-status-map.ts.
  • Webhook delivery loss. Mitigation: nightly reconcile job pulls all Jira deltas.
  • Custom-field maintenance burden. Mitigation: fields managed via Jira REST script-as-code, not console.
