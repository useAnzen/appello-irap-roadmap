1 · Objective

Real construction-domain scenarios are scattered across the codebase: appello/apps/api/prisma/migrations/ (change-order migrations), appello-labs/COMPREHENSIVE_PROPERTY_ANALYSIS.md, appello/plan/clearstory-tm-workorders/. This epic curates them into a benchmark set used to evaluate uis-decomposer and sdlc-auditor on domain-realistic scope (timesheets, change orders, scheduling, T&M work orders). Ian's wording is intentional: this is evaluation corpus, not training data — the agents read it at runtime via retrieval to inform decisions, but no fine-tune.

2 · Acceptance Criteria

  • Corpus ≥ 30 scenarios across 4 domains: timesheets, change orders, scheduling, T&M work orders.
  • Each scenario: { id, domain, narrative, schema_excerpt, user_story, ac, decomposition_oracle, pitfall_notes }.
  • Narrative grounded in real customer evidence (Clearstory T&M, Owens Insulation field notes, etc.) — citations in sources[].
  • Schema excerpts pulled verbatim from appello/apps/api/prisma/schema.prisma + relevant migrations.
  • Decomposition oracle = the expert-validated hamburger slicing (used as the answer key for the eval).
  • Pitfall notes = the specific things the agent has historically gotten wrong (e.g. "forgets Decimal(10,2) for monetary", "merges Create + Update into one hamburger").
  • Corpus checked into agentc2/eval/construction-scenarios/; eval runner integrates with the C1-0034 golden manifest harness.
  • Wording in the eval README explicitly states evaluation corpus, not training data.

3 · Architecture & Approach

Scenario shape

// agentc2/eval/construction-scenarios/timesheets-01.yml
id: timesheets-01
domain: timesheets
narrative: |
  A foreman needs to submit a daily timesheet covering 3 crew members across
  ST/OT/DT hours on two distinct cost codes. Photos and signature required.
schema_excerpt: |
  model Timesheet { id String @id; jobId String; date DateTime; entries TimesheetEntry[] }
  model TimesheetEntry { workerId String; costCode String; stHrs Decimal(5,2); ... }
user_story: |
  As a foreman, I want to submit a multi-worker daily timesheet so payroll runs Monday.
ac: ["Daily total = ST + 1.5*OT + 2*DT", "Signature stored as SVG", "Photo metadata captured", ...]
decomposition_oracle:
  - ts-1-data-model      # Prisma migration + model
  - ts-2-graphql-api     # mutations + queries
  - ts-3-list-view       # day list + filter
  - ts-4-add-edit-form   # multi-worker editor
  - ts-5-signature       # signature capture component
pitfall_notes:
  - "Decomposer often merges form + signature; keep separate."
sources:
  - appello/apps/api/prisma/migrations/2024-09-..-add-timesheet/
  - appello-labs/COMPREHENSIVE_PROPERTY_ANALYSIS.md#timesheets

4 · Implementation Phases (45h)

  1. Source curation across appello/appello-labs/plan repos (10h).
  2. Scenario authoring — 30 yml files across 4 domains (20h).
  3. Decomposition-oracle pairing per scenario (manual expert work) (8h).
  4. Eval runner integration (re-use C1-0034 scoring) (4h).
  5. Doc + clarifying wording around eval-vs-training (3h).

5 · Risks & Mitigations

  • Misread as training corpus by downstream consumers. Mitigation: repeated wording in README + per-file frontmatter purpose: evaluation.
  • Schema excerpts go stale after migrations. Mitigation: CI lint that asserts excerpt source-file SHA matches.
  • Oracle subjectivity. Mitigation: 2-reviewer sign-off per scenario; disagreements escalated.
