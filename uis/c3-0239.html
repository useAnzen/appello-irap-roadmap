1 · Objective

agentc2/tests-e2e/fixtures/ + agentc2/scripts/create-test-tenant.ts + create-test-user.ts seed realistic data; the appello fixture-data files are the template. This epic builds the reusable test-data fixture library so every Spider Swarm run starts from consistent data — same workers, same jobs, same calendar, same material catalog — eliminating flaky tests from data drift.

2 · Acceptance Criteria

  • Fixture pack at appello/tests-e2e/fixtures/data/: workers.json, jobs.json, materials.json, equipment.json, schedules.json, customers.json.
  • seedTenant({ pack, tenantId }) idempotent loader: drops existing rows then inserts pack rows in dependency order.
  • Variant packs: minimal (3 workers, 1 job), typical (15 workers, 10 jobs, full materials), load (100 workers, 80 jobs).
  • Per-spec annotation @dataPack typical picks the right pack at runtime.
  • Fixture-pack snapshot test: bytes-identical hash for each pack version; pinned in CI.
  • Documentation at appello/tests-e2e/fixtures/README.md with per-pack contents + when-to-use.
  • Migration: 15 C3-0232 + 12 C3-0238 specs annotated with their pack.
  • Pack changes go through a versioned migration (pack v2 doesn't silently break v1 specs).

3 · Architecture & Approach

Pack structure

// appello/tests-e2e/fixtures/data/typical/index.ts
export const typicalPack: FixturePack = {
  version: '2026.06.1',
  workers: workers,        // 15 entries
  jobs: jobs,              // 10 entries spanning 3 types
  materials: materials,    // 50 SKUs
  equipment: equipment,    // 12 items
  schedules: schedules,    // next 2 weeks
  customers: customers,    // 6 customers
};

seedTenant

export async function seedTenant({ pack, tenantId }: Args) {
  await db.$transaction(async (tx) => {
    await tx.workerSchedule.deleteMany({ where: { tenantId } });
    // ... cascading deletes ...
    await tx.customer.createMany({ data: pack.customers.map(c => ({ ...c, tenantId })) });
    await tx.worker.createMany(...);
    // ... in FK-order ...
  });
}

5 · Risks & Mitigations

  • Pack creep over time. Mitigation: packs versioned; old packs preserved.
  • Pack-load slow on full reset. Mitigation: snapshot/restore via Postgres template database for large packs.
  • Spec assumes data not in pack. Mitigation: per-spec @dataPack required; CI lint enforces.
