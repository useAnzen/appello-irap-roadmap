1 · Objective

agentc2/docs/sdlc/02-uis-decomposer.md defines the decomposer output schema; agentc2/packages/agentc2/src/tools/json-parser.ts is the AgentC2 utility for structured-output validation; the WorkItem model in appello-sdlc is the persistence target. This epic ties them together: validate decomposer output against a Zod schema, persist into WorkItem rows linked to the parent WorkPackage, tagged with the originating DecompositionVersion (C1-0027).

2 · Acceptance Criteria

  • Zod schema DecomposerOutputSchema mirrors the spec in 02-uis-decomposer.md (hamburgers[], dependencyGraph, coverageReport).
  • persistDecomposerOutput(parentWpId, output, runId) validates, opens a transaction, writes one DecompositionVersion + N WorkItem rows.
  • Each WorkItem persists: slug, sequenceNumber, title, description, scope, ACs, changeSurface, predecessors[], successors[], decompositionVersionId, parentWorkPackageId.
  • Predecessor/successor references resolve to WorkItem.id via a second pass after row insert.
  • Persistence is idempotent on (parentWpId, decomposerRunId): re-running returns the cached DecompositionVersion.
  • Schema-validation failure produces a structured error logged to AgentLearning so the agent can self-correct.
  • End-to-end smoke: run decomposer on UIS-20 sample → 8 WorkItem rows written; predecessors resolve correctly.

3 · Architecture & Approach

Persistence flow

// agentc2/packages/agentc2/src/persistence/decomposer.ts
import { DecomposerOutputSchema } from './schemas';

export async function persistDecomposerOutput(parentWpId: string, raw: unknown, runId: string) {
  const out = DecomposerOutputSchema.parse(raw);  // throws on invalid
  return await db.$transaction(async (tx) => {
    const version = await tx.decompositionVersion.upsert({
      where:  { uisId_runId: { uisId: parentWpId, runId } },
      create: { uisId: parentWpId, runId, hamburgersJson: out.hamburgers, version: await nextVersion(parentWpId, tx) },
      update: {}
    });
    if (version.workItems.length) return version;  // already populated
    const rows = await Promise.all(out.hamburgers.map((h, i) =>
      tx.workItem.create({ data: {
        slug: h.slug, sequenceNumber: i+1, title: h.title, description: h.description,
        scope: h.scope, ac: h.acceptanceCriteria, changeSurface: h.changeSurface,
        parentWorkPackageId: parentWpId, decompositionVersionId: version.id,
      } })
    ));
    await resolvePredecessorRefs(rows, out, tx);
    return version;
  });
}

4 · Implementation Phases (15h)

  1. Zod schema authoring (3h).
  2. persistDecomposerOutput + transactional writes (5h).
  3. Predecessor/successor resolution pass (3h).
  4. Idempotency cache key + dedup (2h).
  5. End-to-end smoke against UIS-20 sample (2h).

5 · Risks & Mitigations

  • Predecessor cycles unprovable until both ends inserted. Mitigation: two-pass insert; reject on cycle after pass 2.
  • Large hamburger arrays exceed transaction time. Mitigation: chunk insert (50 per chunk) within outer tx.
  • Schema-drift between agent prompt and Zod schema. Mitigation: single source: prompt references schema's .describe() strings.
