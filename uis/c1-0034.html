1 · Objective

Author a frozen library of human-validated golden hamburgers — reference decompositions that the uis-decomposer agent (C1-0029) is scored against. Each entry is a full hamburger-manifest-*.json describing how an approved 8-section UIS was sliced into dependency-ordered, individually mergeable hamburgers with change surface, complexity score, and acceptance criteria. The set seeds the eval rubric used to drive the Decomposer's 22% success rate upward.

2 · Acceptance Criteria

  • ≥ 2 full reference manifests checked into appello-roadmap/hamburger-manifest*.json, each derived from a real merged UIS.
  • Every hamburger entry carries the canonical fields: slug, sequenceNumber, title, description, predecessors[], hasUiInterfaces, complexityScore, estimatedDays, changeSurface{new[],modified[],deleted[]}, acceptanceCriteria[], uis.
  • Dependency graph is acyclic; every predecessors reference resolves to a sibling slug in the same manifest.
  • Change surfaces are real file paths in useAnzen/application-mono-repo (apps/api, apps/frontend) — not invented.
  • Coverage check: union of all hamburgers covers the full parent UIS scope without orphan acceptance criteria.
  • Manifests are loaded as test fixtures by the decomposer eval harness; pass/fail score computed per hamburger.

4 · Reference Benchmark — UIS-20 T&M Work Orders

Real parent UIS: uis-20-tm-work-orders.html · Total complexity: 71 · Total estimated days: 40 · Prerequisite: Emma's Job DataModel refactor merged.

WO-1: Job Type-Specific Configuration

slug: wo-1-job-type-config · seq 1 · complexity 7 · ~4d · predecessors: []

Description: Extend the Job DataModel so different jobType values trigger different form configurations, status workflows, and UI behavior. Add isWorkOrder flag, WO-specific PropertyValue seeds, and DataModel form configurations.

Change surface — new: apps/api/prisma/migrations/YYYYMMDDHHMMSS_add_work_order_fields/migration.sql

Change surface — modified: apps/api/prisma/schema.prisma apps/api/src/models/Job.ts apps/api/src/graphql/models/Job.ts apps/frontend/src/pages/jobs/index.tsx apps/frontend/src/components/Forms/JobAddEditForm/JobDataModelForm.tsx

Acceptance criteria (selected):

  • JobType "T&M Work Order" available in PropertyValue seed data
  • Selecting WO type on job creation sets isWorkOrder = true on the Job record
  • WO-specific status workflow loaded: Draft → Submitted → Awaiting Signature → Signed → Approved → Invoiced → Void
  • Job list page has filter/toggle for All Jobs / Project Jobs / Work Orders
  • Work Orders can be created without a project (projectId optional)

WO-2: Work Order Line Item Tables

slug: wo-2-line-item-tables · seq 2 · complexity 11 · ~5d · predecessors: [wo-1-job-type-config]

Description: New Prisma models for structured labor, material, and equipment line items on Work Order jobs, with auto-pricing from existing rate tables (TradeLevel, Material, Equipment catalogs).

Change surface — new (Prisma + GraphQL + UI): apps/api/src/models/WorkOrderLaborItem.ts apps/api/src/models/WorkOrderMaterialItem.ts apps/api/src/models/WorkOrderEquipmentItem.ts apps/api/src/graphql/models/WorkOrderLineItems.ts apps/frontend/src/components/Forms/WorkOrderLineItems/WorkOrderLaborItemEditor.tsx apps/frontend/src/components/Forms/WorkOrderLineItems/WorkOrderMaterialItemEditor.tsx apps/frontend/src/components/Forms/WorkOrderLineItems/WorkOrderEquipmentItemEditor.tsx apps/frontend/src/graphql/queries/workOrderLineItems.graphql apps/frontend/src/graphql/mutations/workOrderLineItems.graphql

Acceptance criteria (selected):

  • Labor line item has trade-level combobox, description, ST/OT/DT hours, auto-populated rate, auto-calculated total
  • Changing trade level auto-populates base rate, OT rate (1.5×), DT rate (2×) from TradeLevel + WageRegion
  • Material line item has material combobox, description, quantity, auto-populated unit cost, auto-calculated total
  • Equipment line item has equipment combobox, description, hours, auto-populated rate, auto-calculated total
  • Line items are reorderable with drag handles and persisted orderNo
  • All monetary values stored as Decimal(10,2)

Remaining hamburgers in this manifest (slugs only — full ACs in the JSON): wo-3-signature-capture, wo-4-status-workflow, wo-5-change-order-log, wo-6-invoicing-handoff, wo-7-pdf-export, wo-8-history-audit (see appello-roadmap/hamburger-manifest.json for the full set).

5 · Reference Benchmark — UIS-1.34 (second manifest)

A second manifest is published at appello-roadmap/hamburger-manifest-1.34.json against the 1.34 release UIS. It diversifies the benchmark set across:

  • A data-model-heavy feature (schema migrations dominate change surface)
  • A UI-heavy feature (frontend components dominate)
  • A mixed feature with cross-package effects (API + frontend + worker)

This diversity prevents the decomposer from overfitting to a single decomposition shape and is what allows the eval harness to surface failure-mode regressions when the agent's prompt or model changes.

6 · How the Eval Harness Consumes These

  1. Decomposer is run against the parent UIS only (the manifest is hidden).
  2. Its output is structurally diffed against the golden manifest:
    • Hamburger count parity (±1 tolerated, ±2 fails)
    • Predecessor graph topology match (DAG isomorphism with relaxed labelling)
    • Change-surface coverage: union of agent's file lists must cover ≥ 90% of golden union
    • Acceptance-criteria recall against the golden ACs (string-similarity + LLM judge)
  3. Each hamburger receives a pass / partial / fail; the manifest receives a weighted score.
  4. Scores feed the decomposer success-rate metric on the overview page (currently 22%, target ≥ 70% — see C1-0029 backlog).

7 · Implementation Phases (32h)

  1. UIS-20 manifest authoring (12h) — slice T&M Work Orders into 8 hamburgers, write descriptions/ACs/change surfaces from the actual merged PRs.
  2. 1.34 manifest authoring (10h) — second reference covering a different decomposition shape.
  3. Schema validation tooling (4h) — JSON schema for hamburger entries; CI lint to fail on missing fields / cyclic dependencies / phantom predecessors.
  4. Eval harness integration (4h) — wire manifests as fixtures into the decomposer's scoring run.
  5. Review + sign-off (2h) — human verification that manifests match what actually shipped.

8 · Risks & Mitigations

  • Benchmark drift after refactors — when apps/api/src/models/Job.ts moves, change-surface paths in the manifest go stale. Mitigation: CI job that verifies every changeSurface path exists in the current monorepo tree.
  • Overfitting to two examples — decomposer learns the shape of WO/1.34 specifically. Mitigation: rotate in new manifests each release; keep older manifests as regression tests.
  • Subjective coverage scoring — "covers ≥ 90% of golden union" leaves room for judgement. Mitigation: dual scorer (deterministic file-path overlap + LLM-judge for semantic gaps); flag low-confidence cases for human review.
  • Author bias — author of the golden manifest also tunes the decomposer prompt. Mitigation: hold out one manifest from the prompt-iteration loop; use only for final eval.
