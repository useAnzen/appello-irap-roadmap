1 · Objective

The uis-decomposer is live (v3, 8 steps, 9 historical runs at 22% success per agentc2/docs/sdlc/00-overview.md); the Vitest pattern is established in agentc2/tests/unit/inngest-functions.test.ts. This epic adds the validation layer that hardens decomposer output against its known failure modes: subtasks must be actionable (have a clear deliverable), correctly scoped (≤5 stories / ≤15 ACs / ≥2 ACs per hamburger), and complete (every parent AC mapped). Tests gate the decomposer's promotion to autonomous dispatch.

2 · Acceptance Criteria

  • Test file tests/integration/uis-decomposer.quality.test.ts runs the decomposer against the C1-0034 golden manifests + the C1-0019 construction-scenario corpus.
  • Actionability check: every hamburger has ≥ 1 verb-led deliverable in its description (regex: /(Build|Add|Create|Refactor|Wire|Migrate|...)/i at sentence start).
  • Scope-bounds check: every hamburger has ≥ 2 ACs, ≤ 15 ACs, ≤ 5 user stories.
  • Coverage check: union of all hamburger ACs = parent UIS ACs (no gaps, no duplicates).
  • Dependency-acyclicity check: topological sort succeeds on the dependency graph.
  • Sequencing check: data-model hamburgers precede API hamburgers; API precedes UI (per the decomposer's own principles in C1-0029).
  • Score is reported as a percentage of checks passed; CI threshold = 80% to allow autonomous dispatch.
  • Failing checks emit structured diagnostics that feed back into the decomposer's prompt (next-run context).

3 · Architecture & Approach

Test harness

// tests/integration/uis-decomposer.quality.test.ts
const goldens = await loadGoldens('appello-roadmap/hamburger-manifest*.json');
const scenarios = await loadScenarios('agentc2/eval/construction-scenarios/');

describe.each([...goldens, ...scenarios])('decomposer quality / $id', (case) => {
  let output: DecomposerOutput;
  beforeAll(async () => { output = await runDecomposer(case.parentUis); });

  it('every hamburger is actionable', () => expect(checkActionability(output)).toBe(true));
  it('every hamburger is scope-bounded', () => expect(checkScopeBounds(output)).toEqual([]));
  it('union of ACs covers parent UIS', () => expect(checkAcCoverage(output, case)).toEqual({ gaps: [], duplicates: [] }));
  it('dependency graph is acyclic', () => expect(topoSort(output.dependencyGraph)).not.toThrow());
  it('sequencing follows principles', () => expect(checkSequencing(output)).toEqual([]));
});

Diagnostic feedback loop

// When a check fails, the diagnostic is written to the decomposer's prompt-grounding store
await agentLearning.upsert({
  agentSlug: 'uis-decomposer',
  topic: 'common-mistakes',
  body: `On case ${case.id}, you produced a hamburger with no actionable verb in description: "${hamburger.description}". Always start with Build/Add/Wire/Refactor.`
});

4 · Implementation Phases (50h)

  1. Test harness scaffolding (6h).
  2. Actionability check + regex tuning (5h).
  3. Scope-bounds + coverage checks (10h).
  4. Dependency acyclicity + sequencing checks (8h).
  5. Score aggregation + CI threshold (4h).
  6. Diagnostic feedback into agent-learning store (5h).
  7. Run against full golden+scenario set; tune checks (10h).
  8. Doc + runbook (2h).

5 · Risks & Mitigations

  • Regex actionability check has false negatives (legitimate non-verb starts). Mitigation: hand-reviewed allow-list; quarterly false-negative audit.
  • Tests pass on goldens, fail on real new UISs. Mitigation: the construction scenarios provide novelty; CI also runs on random sample of recent UISs.
  • Decomposer over-optimizes to pass checks rather than produce useful slicings. Mitigation: hold one golden out of the prompt-grounding loop as a true holdout.
