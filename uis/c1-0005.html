1 · Objective

The appello-sdlc Next.js frontend already renders work packages, stage outputs, reviews, and the enrichment timeline (see screenshots 01-work-packages-list-current-state through 10-* in the repo root). This epic adds a live overlay across that board: for every active work package, show which of the 7 SDLC stages (Triage → UIS Gen → Decompose → Plan → Build → Review → Merge) currently has an agent working on it, what the agent is doing, and the per-stage SLA clock. The overlay turns the existing board from a record-of-truth into an operational console.

2 · Acceptance Criteria

  • Each work-package card shows a 7-stage horizontal progress strip; current stage highlighted; completed stages green; failed stages red.
  • Hovering the current stage shows the live agent (slug + model), the latest stage_outputs line, and elapsed time since stage entry.
  • Stage transitions are sourced from transition_logs (C1-0001); live agent activity sourced from AgentRun + stage_outputs tail.
  • Updates stream via Server-Sent Events from /api/stream/pipeline; latency p95 < 2s from DB write to UI.
  • Per-stage SLA defined per work-package type (default 6h Triage / 30 min others); badge turns amber at 50% SLA, red at 100%.
  • Aggregate counters at top: # active by stage; weekly throughput; mean stage latency.
  • Mobile-responsive: strip collapses to vertical accordion below 720px.

3 · Architecture & Approach

Data flow

transition_logs (write)  ──┐
stage_outputs (append)   ──┼──> Postgres LISTEN/NOTIFY ──> Next.js SSE ──> React board
AgentRun heartbeat       ──┘

Component layout

  • apps/frontend/src/app/pipeline/_components/StageStrip.tsx (new) — 7-segment strip + tooltip.
  • apps/frontend/src/app/pipeline/_components/LiveActivityProvider.tsx (new) — SSE subscription, distributes updates by workPackageId.
  • apps/frontend/src/app/api/stream/pipeline/route.ts (new) — SSE endpoint multiplexing pg-notify events.
  • apps/frontend/src/app/pipeline/page.tsx (modified) — wraps existing list in LiveActivityProvider.

Stage model

type Stage = 'triage'|'uis-gen'|'decompose'|'plan'|'build'|'review'|'merge';
const SLA_HOURS: Record<Stage, number> = {
  triage: 6, 'uis-gen': 4, decompose: 1, plan: 1, build: 2, review: 1, merge: 0.5,
};

4 · Implementation Phases (55h)

  1. Stage-strip component + states (10h) — pure component with Storybook stories for all 7×3 combinations.
  2. LIVE activity provider + SSE plumbing (15h) — Postgres LISTEN/NOTIFY → Next.js Edge runtime SSE endpoint.
  3. Aggregate counters + SLA badges (8h) — top bar + per-card color rules.
  4. Mobile responsive + accessibility pass (6h).
  5. Integration with existing board + screenshot regression suite (10h).
  6. Soak test (6h) — 200 simulated work packages, 50 stage transitions/min for 2h; assert no SSE backpressure, p95 < 2s.

5 · Risks & Mitigations

  • SSE connection churn at scale. Mitigation: single SSE per browser, fan-out via in-memory broker; LISTEN/NOTIFY on Postgres is per-connection so use one shared listener.
  • Stage taxonomy drift if pipeline adds an 8th stage. Mitigation: stages enum is in shared package; CI test prevents UI/DB divergence.
  • Tooltip data leaks (showing PII from stage_outputs). Mitigation: server-side scrubber on the SSE feed.
