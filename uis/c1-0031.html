1 · Objective

appello/packages/shared-utils/hasPermissions.ts is the runtime guard against the 61-domain PermissionItemsEnum; agentc2/packages/agentc2/src/tools/verify-tools.ts is where auditor rules attach. sdlc-auditor exists. This epic codifies the Appello-specific checks (correct GraphQL resolver patterns, Prisma migration safety, RBAC guard usage, no destructive SQL) into the auditor's rule set so the same patterns the human team enforces become automatic.

2 · Acceptance Criteria

  • Rule registry at agentc2/packages/agentc2/src/rules/appello/ with one file per rule.
  • Rule 1 — resolver-pattern: every new GraphQL resolver follows { Query|Mutation }.<name> = withAuth(handler) shape.
  • Rule 2 — prisma-migration-safety: no DROP COLUMN / DROP TABLE in a single migration without companion idempotent SQL (per appello-xrm-prisma-migrations).
  • Rule 3 — rbac-guard: every mutation calls hasPermissions(ctx.user, [PermissionItem.X]) before any DB write.
  • Rule 4 — no-raw-destructive: prisma.$executeRaw with DROP/DELETE/UPDATE rejected unless explicitly annotated // @audit-allow-destructive: <reason>.
  • Rule 5 — decimal-money: monetary fields use Decimal(10,2); Float on a monetary field = audit fail.
  • Each rule emits structured findings: { ruleId, severity, file, line, message, suggestion }.
  • Findings flow into Review.auditFindings Json column; audit fail = trust score -25 per critical.

3 · Architecture & Approach

Rule shape

// agentc2/packages/agentc2/src/rules/appello/rbac-guard.ts
import type { AuditRule, AuditFinding } from '../types';
export const rbacGuard: AuditRule = {
  id: 'appello/rbac-guard',
  severity: 'critical',
  appliesTo: (file) => file.endsWith('.resolvers.ts'),
  check: async (file, diff) => {
    const findings: AuditFinding[] = [];
    for (const fn of extractMutations(file)) {
      if (!callsHasPermissions(fn)) {
        findings.push({ ruleId: 'appello/rbac-guard', severity: 'critical',
          file: file.path, line: fn.line,
          message: `Mutation ${fn.name} missing hasPermissions guard`,
          suggestion: `Wrap with hasPermissions(ctx.user, [PermissionItem.X])` });
      }
    }
    return findings;
  }
};

4 · Implementation Phases (25h)

  1. Rule registry scaffolding (3h).
  2. Authoring rules 1-5 (15h, ~3h each).
  3. Integration into verify-tools dispatch (3h).
  4. Review.auditFindings persistence + UI surfacing (2h).
  5. End-to-end test against last 10 merged PRs to validate signal-to-noise (2h).

5 · Risks & Mitigations

  • False positives on legitimate patterns (e.g. background jobs). Mitigation: per-rule allow-list config; reviewers can append justified exceptions.
  • Rule drift from real Appello conventions. Mitigation: rules co-located with appello package they enforce; PRs to appello touching guards also touch rule.
  • Performance on large diffs. Mitigation: rules scoped to changed files; AST cached per file SHA.
