1 · Objective

Define the six-agent SDLC pipeline that turns a single GitHub issue (or other ticket source) into a merged pull request without a human in the loop on any mechanical step. Each agent has a narrow mission, a typed I/O contract, a guardrail, an escalation rule, and is registered in the AgentC2 database by a single idempotent seed script (scripts/seed-sdlc-playbook.ts). The whole flow is orchestrated by the cron-driven sdlc-pipeline-supervisor Inngest function so that no human babysits state transitions between agents.

The deliverable for C1-0015 is the roster + handoff contract: the agent specs, the JSON schemas that flow between them, the supervisor's chain rules, and the playbook seed that materializes them all in the database. The runtime that actually executes the workflows is shipped in C1-0003; the schema that stores their state is C1-0001.

2 · Acceptance Criteria

  1. Six pipeline roles are specified and named: sdlc-classifier, sdlc-planner, cursor-launch-agent (implementer tool wrapping the Cursor Cloud Agent API), sdlc-auditor, sdlc-reviewer, and sdlc-pipeline-supervisor.
  2. Each LLM agent has a frozen system prompt, a typed JSON output contract, a model/temperature/maxSteps choice, an attached scorecard with weighted criteria, ≥ 1 test case, and a guardrail policy (blocked tools, max tokens, human approval flag).
  3. The handoff path is explicit and reproducible: classifier → planner ↔ auditor (revision loop) → cursor-launch-agent (implement) → reviewer → merge gate. The supervisor's chain rules link the workflow runs in DB so a partial pipeline can be resumed from any step.
  4. Escalation paths are defined for every agent (what it does on FAIL/BLOCK/NEEDS_REVISION/timeout) — see §7 per-agent specs.
  5. JSON contracts for classifier and auditor are validated at runtime by jsonParserTool (packages/agentc2/src/tools/json-parser.ts) so a malformed agent response fails fast instead of corrupting downstream state.
  6. The playbook seed (scripts/seed-sdlc-playbook.ts) creates the four DB-resident agents, three SDLC workflows (sdlc-triage, sdlc-bugfix, sdlc-feature), the entry webhook trigger, and the published sdlc-flywheel playbook — idempotently — in a single bun run invocation.
  7. The supervisor function (apps/agent/src/lib/sdlc-pipeline-supervisor.ts) runs on a 10-minute cron, chains uis-generator → uis-decomposer → hamburger-batch-execute, and reports stuck runs (suspended > 30 min or RUNNING > 4 h) for operator visibility.

3 · Agent roster — the six contracted roles

Four agents are LLM agents persisted in the Agent table via the playbook seed (C1-0001 schema). One is a Mastra tool that wraps the Cursor Cloud Agent REST API and acts as the implementer (it does not need its own LLM agent row — the LLM lives at Cursor's end). The sixth is a cron-driven Inngest function, not an LLM.

Why a tool instead of a sixth LLM agent for the implementer? The implementation step needs full repo read/write access, which Cursor Cloud Agents already provide via the /v0/agents API. Wrapping that as a Mastra tool keeps the contract narrow (just a JSON request) and avoids running a second LLM in our process when Cursor is already running one for us. The role in the pipeline is still “implementer agent” — it just happens to be hosted by Cursor.

5 · Per-agent specs — mission, I/O, escalation

5.1 sdlc-classifier DONE openai/gpt-4o · T=0.3

Mission: Read an incoming ticket and decide what kind of work it is, how urgent, how big, and which downstream workflow should handle it. The classifier never reads code — it works entirely from the ticket title + description + repository tag.

Source: scripts/seed-sdlc-playbook.ts:87-104 (instructions), :445-499 (agent def)

// Input — the ticket as it arrives from the webhook trigger
{
    title: string,
    description: string,
    repository: string     // e.g. "useAnzen/agentc2"
}

// Output — strict JSON, validated by json-parser before routing
{
    classification: "bug" | "feature" | "user_error" | "documentation" | "infrastructure",
    priority:       "trivial" | "low" | "medium" | "high" | "critical",
    complexity:     "trivial" | "low" | "medium" | "high" | "critical",
    affectedAreas:  string[],
    suggestedRoute: "sdlc-bugfix" | "sdlc-feature" | "sdlc-triage",
    rationale:      string
}
Escalation: If classification === "user_error" the triage workflow short-circuits — the planner writes a KB article in place of the bug/feature workflow. Any other value routes to the matching sub-workflow via the route branch step.

5.2 sdlc-planner DONE anthropic/claude-sonnet-4-6 · T=0.5

Mission: Turn a Cursor Cloud Agent codebase analysis into an implementation plan with file-level specificity. The planner deliberately does not read the codebase itself — Cursor Cloud has full repo access and produced the analysis; the planner's job is to convert that analysis into a sequenced plan.

Source: scripts/seed-sdlc-playbook.ts:27-42 (instructions), :500-548 (agent def)

// Input — assembled by the workflow's promptTemplate
{
    analysisText:     string,      // from cursor-launch-agent (analysis run)
    ticketTitle:      string,
    classification:   string,
    priorAuditNotes?: string,      // only on revision iterations
    humanFeedback?:   string       // only after human plan-review
}

// Output — free-form markdown (planner is not constrained to JSON)
// Must contain: numbered steps, file paths, complexity per step,
// risks/dependencies, testing notes per change.
Escalation: Plan is always sent to sdlc-auditor next. If auditor returns NEEDS_REVISION, the workflow's plan-cycle dowhile (max 3 iterations) re-invokes the planner with audit feedback in priorAuditNotes. A 4th failed audit escalates to human via the plan-review human step.

5.3 cursor-launch-agent DONE Mastra tool · wraps api.cursor.com/v0

Mission: The implementer. Hand it a repo + a detailed prompt and it spawns a Cursor Cloud Background Agent that clones the repo, writes code, and pushes a branch (optionally opening a PR). Used three times in the bugfix/feature workflows: (1) analysis-only run before planning, (2) implementation run after the plan is approved, (3) self-heal run when CI fails.

Source: packages/agentc2/src/tools/cursor-tools.ts:100-180 (tool def), :22-54 (credential resolution)

export const cursorLaunchAgentTool = createTool({
    id: "cursor-launch-agent",
    description: "Launch a Cursor Cloud Agent to implement code changes on a "
        + "GitHub repository.",
    inputSchema: z.object({
        repository: z.string(),                  // "https://github.com/org/repo"
        prompt:     z.string(),                  // detailed task
        ref:        z.string().optional(),       // default "main"
        model:      z.string().optional(),       // "claude-4.6-opus-high-thinking"
        branch:     z.string().optional(),       // reuse a branch on self-heal
        autoCreatePr: z.boolean().optional()
    }),
    // outputs: { agentId, status, branchName?, prUrl? }
});
Escalation: Paired with cursor-poll-until-done (15-min cap for analysis, 30-min cap for implementation, 15-min cap for self-heal). On poll timeout the workflow's heal-cycle dowhile re-launches with the failure log appended to the prompt; after 3 failed self-heal iterations the run is marked failed and surfaced via supervisor stuck-run reporting.

5.4 sdlc-auditor DONE anthropic/claude-sonnet-4-6 · T=0.3

Mission: Quality gate. Reads a plan or an analysis or a diff and decides if it can proceed. Returns strict JSON with a four-state verdict and a structured issue list so the workflow can route mechanically without re-parsing prose.

Source: scripts/seed-sdlc-playbook.ts:44-85 (instructions), :549-597 (agent def)

{
    verdict: "PASS" | "APPROVED_WITH_NOTES" | "NEEDS_REVISION" | "FAIL",
    severity: "none" | "minor" | "major" | "critical",
    issues: [
        { severity: "critical|major|minor",
          area: string, description: string, suggestedFix: string }
    ],
    positives: string[],
    summary: string,
    checklist: {
        requirementsAddressed: boolean,
        edgeCasesConsidered:   boolean,
        errorHandlingPresent:  boolean,
        noBreakingChanges:     boolean,
        securityReviewed:      boolean,
        performanceAssessed:   boolean,
        testingCovered:        boolean
    }
}
Escalation: The verdict-route branch decides the next step: PASS / APPROVED_WITH_NOTES → human plan-review; NEEDS_REVISION → loop back to planner with feedback posted to the GitHub issue; FAIL → halt and escalate. The auditor is blocked from calling cursor-launch-agent or merge-pull-request directly (blockedTools guardrail).

5.5 sdlc-reviewer DONE anthropic/claude-sonnet-4-6 · T=0.3 · human-approval required

Mission: PR review with a calibrated trust score. Reads the diff, the CI results, and any prior review depth, then emits a 0–100 score broken into four 25-point sub-scores plus an APPROVE / REQUEST_CHANGES / BLOCK recommendation.

Source: scripts/seed-sdlc-playbook.ts:106-130 (instructions), :598-646 (agent def)

{
    recommendation: "APPROVE" | "REQUEST_CHANGES" | "BLOCK",
    trustScore: 0–100,
    breakdown: {
        testCoverage:   0–25,
        codeQuality:    0–25,
        reviewDepth:    0–25,
        riskAssessment: 0–25
    },
    issues: [{ severity: "critical|major|minor", description: string }],
    summary: string
}
Escalation: Reviewer's guardrail sets requireHumanApproval: true — its merge recommendations are advisory, never automatic. Production-target merges always require a human signoff. REQUEST_CHANGES drops back into the self-heal dowhile with the reviewer's issues array as the next prompt input.

5.6 sdlc-pipeline-supervisor DONE Inngest function · cron */10 * * * *

Mission: The coordinator. Every 10 minutes, walks the latest WorkflowRun for the three SDLC workflows (uis-generator, uis-decomposer, hamburger-factory), chains a COMPLETED upstream run's output into a QUEUED downstream run, and emits batch events for the hamburger factory. Also flags stuck runs.

Source: apps/agent/src/lib/sdlc-pipeline-supervisor.ts:22-411 (full function)

// Inputs: none — reads workflow rows by slug
const SDLC_WORKFLOW_SLUGS = {
    uisGenerator:     "uis-generator",
    uisDecomposer:    "uis-decomposer",
    hamburgerFactory: "hamburger-factory"
};

// Side effects:
// 1. uis-generator COMPLETED → prisma.workflowRun.create({ uis-decomposer, QUEUED })
//    + inngest.send("workflow/execute.async", { workflowRunId, ... })
// 2. uis-decomposer COMPLETED → inngest.send("hamburger/batch.execute", {
//        batchId, uisId, parentSlug, featureWorkflowSlug, manifest, organizationId })
// 3. stuck detection: RUNNING > 4h OR (suspendedStep && elapsed > 30min)
//    → returned as `stuckRuns: string[]` for operator visibility
Escalation: The supervisor itself does not escalate — it reports. Stuck runs are surfaced in its return value and console log. Idempotency is enforced by a check-X-consumed step that asks prisma.workflowRun.findFirst({ inputJson: { path: ["_sourceRunId"], equals: latestRuns.X.id } }) before queuing the downstream run, so re-running the cron over the same input is a no-op.

6 · Supervisor contract — how the chain rules actually work (real code)

The supervisor is the only piece of orchestration that lives outside the workflow engine — it is a plain Inngest cron function. Its handoff contract is just “upstream outputJson shape must contain the key that downstream expects”. Below is the verbatim generator → decomposer chain.

Source: apps/agent/src/lib/sdlc-pipeline-supervisor.ts:107-164

// Chain: uis-generator COMPLETED → invoke uis-decomposer.
// Uses source metadata to prevent duplicate chaining across supervisor cycles.
if (latestRuns.uisGenerator?.status === "COMPLETED") {
    const output = latestRuns.uisGenerator.outputJson as Record<string, unknown> | null;
    const decomposerInput = output?.decomposerInput as Record<string, unknown> | null;

    if (decomposerInput) {
        const alreadyChained = await step.run("check-generator-consumed", async () => {
            const existing = await prisma.workflowRun.findFirst({
                where: {
                    workflowId: workflows.uisDecomposer!.id,
                    source: "pipeline-supervisor",
                    inputJson: {
                        path: ["_sourceRunId"],
                        equals: latestRuns.uisGenerator!.id
                    }
                },
                select: { id: true }
            });
            return !!existing;
        });

        if (!alreadyChained) {
            await step.run("chain-generator-to-decomposer", async () => {
                const wf = workflows.uisDecomposer!;
                const enrichedInput = {
                    ...decomposerInput,
                    _sourceRunId: latestRuns.uisGenerator!.id
                };
                const runRecord = await prisma.workflowRun.create({
                    data: {
                        workflowId: wf.id,
                        status: "QUEUED",
                        inputJson: enrichedInput,
                        environment: "PRODUCTION",
                        triggerType: "SCHEDULED",
                        source: "pipeline-supervisor"
                    }
                });

                await inngest.send({
                    name: "workflow/execute.async",
                    data: {
                        workflowRunId: runRecord.id,
                        workflowId:    wf.id,
                        workflowSlug:  wf.slug,
                        input:         enrichedInput
                    }
                });
            });
        }
    }
}

The _sourceRunId tag is the linchpin of the contract: it makes the chain step idempotent against the cron's at-least-once delivery, and it gives every downstream run a back-pointer to the upstream that birthed it — used by stuck-run diagnostics and by the C1-0029 decomposer to reassemble manifest provenance.

7 · Playbook seed — how the roster lands in the database

A single bun run scripts/seed-sdlc-playbook.ts creates (or updates) all four LLM agents, attaches their skills, scorecards, test cases, guardrails, and memory configs, then registers the workflows and the entry webhook trigger. The script is idempotent — it checks for the agent by slug and skips the create if it exists. Below is the verbatim agent-create block.

Source: scripts/seed-sdlc-playbook.ts:679-768

const agents: Record<string, { id: string; slug: string }> = {};
for (const agentDef of agentDefs) {
    let existing = await prisma.agent.findFirst({
        where: { slug: agentDef.slug }
    });
    if (!existing) {
        const agentRole = agentDef.slug.includes("classifier") ? "classifier"
            : agentDef.slug.includes("planner")  ? "planner"
            : agentDef.slug.includes("auditor")  ? "auditor"
            : "reviewer";

        existing = await prisma.agent.create({
            data: {
                slug:           agentDef.slug,
                name:           agentDef.name,
                description:    agentDef.description,
                instructions:   agentDef.instructions,
                modelProvider:  agentDef.modelProvider,
                modelName:      agentDef.modelName,
                temperature:    agentDef.temperature,
                maxSteps:       agentDef.maxSteps,
                memoryEnabled:  true,
                memoryConfig:   memoryConfigs[agentRole],
                contextConfig:  contextConfigs[agentRole],
                visibility:     "ORGANIZATION",
                workspaceId:    workspace.id
            }
        });

        // Attach tools, skills, scorecard, test cases, guardrail (idempotent inserts)
        for (const toolId   of agentDef.tools)   await prisma.agentTool.create({...});
        for (const skillSlug of agentDef.skills) await prisma.agentSkill.create({...});
        if (agentDef.scorecard) await prisma.agentScorecard.create({ data: { agentId: existing.id, criteria: agentDef.scorecard.criteria } });
        for (const tc of agentDef.testCases) await prisma.agentTestCase.create({ data: { agentId: existing.id, ...tc, tags: ["sdlc"] } });
        if (agentDef.guardrail) await prisma.guardrailPolicy.create({ data: { agentId: existing.id, configJson: agentDef.guardrail } });
    }
    agents[agentDef.slug] = { id: existing.id, slug: existing.slug };
}

After the agents are in place the same script creates the three workflows (sdlc-triage, sdlc-bugfix, sdlc-feature) by upserting their definitionJson directly, then packages the whole bundle into the published sdlc-flywheel playbook via packagePlaybook() from packages/agentc2/src/playbooks/packager.ts. A re-run of the seed is detected by slug and downgrades to workflow.update + console.log("Updated workflow:", existing.slug) — no duplicates, no drift.

Verification. After seeding, the AgentC2 admin UI shows four DB rows under Agents (sdlc-classifier-agentc2, sdlc-planner-agentc2, sdlc-auditor-agentc2, sdlc-reviewer-agentc2) each with attached scorecard, ≥ 1 test case, and a guardrail policy. The published playbook is browseable at /playbooks/sdlc-flywheel and installable into any organization with the GitHub + Cursor integrations connected.

9 · Test plan

9.1 Each agent runs standalone

Every agent in §3 is invocable via the platform's agent-invoke surface with its own test case from the seed:

// scripts/seed-sdlc-playbook.ts — every agent has testCases[]
// Run: bun run agent.invoke({ slug: "sdlc-classifier-agentc2",
//        inputText: "The login button does not respond on mobile Safari…" })
// Expected: { classification: "bug", priority: "high", ... }

Each agent's scorecard runs against its test cases on every CI pipeline; a regression in classification accuracy or audit verdict precision blocks the deploy.

9.2 Handoff round-trip

  1. POST a synthetic issue.labeled webhook to the trigger; verify sdlc-triage run starts.
  2. Watch the workflow run-steps table — there must be one WorkflowRunStep row per agent invocation in the order specified by §4.
  3. Re-run the same webhook payload; verify the check-X-consumed idempotency guard fires and no duplicate run is created.

9.3 Supervisor failure mode

  • Kill an in-flight sdlc-feature run while it is suspended at the human plan-review step.
  • Wait 31 minutes; the next supervisor cron tick must include that run in stuckRuns[] with the message "…: suspended at step 'plan-review'".
  • Manually re-emit the human approval event; verify the workflow resumes and the supervisor does not double-chain its downstream run (idempotency check holds).

9.4 Contract drift

Inject a malformed JSON output from the auditor (simulated by overriding the agent response in a test harness). Expected behavior: jsonParserTool throws Invalid JSON: …, the workflow's plan-verdict-route branch defaults safely to NEEDS_REVISION, and the dowhile re-invokes the planner. No downstream step runs against a broken contract.

10 · Risks & mitigations

RiskImpactMitigation
Contract drift across deploys — a future change to the auditor prompt adds a new verdict value (e.g. NEEDS_MINOR_TWEAK) and the workflow's verdict-route branch silently falls through to the default arm. High JSON shapes are duplicated in two places by design: the agent prompt itself (§5.4) and the jsonParserTool consumer. CI lint rule (TODO follow-up epic) will assert prompt-string membership in the consumer's allowed-value set; until then the auditor's prompt explicitly enumerates the four legal verdicts so any change requires editing both ends.
Supervisor SPOF — the entire pipeline depends on one Inngest cron firing every 10 minutes. If Inngest goes down or the function errors silently, chained runs stall. High Three guards: (a) Inngest retries the function { retries: 1 } on failure; (b) supervisor returns a structured report with timestamp + actions + stuckRuns so operator dashboards can alert on missing reports; (c) the function is stateless — re-running it after an outage simply chains any COMPLETED upstream runs the previous tick missed (idempotency holds via _sourceRunId).
Agent prompt evolution — an instruction change to the planner subtly shifts its plan format and the auditor starts failing every plan because it can no longer find the “numbered steps” checklist item. Med Every agent prompt change goes through seed-sdlc-playbook.ts (single source of truth) and is review-gated. The scorecard's test cases run against the new prompt on every deploy — a regression in “Plan Quality” score below the previous baseline blocks the playbook publish.
Cursor Cloud Agent API outage — the implementer is hosted by Cursor; if their API is down or rate-limited, the entire pipeline halts at the implement-launch step. Med cursorFetch() in cursor-tools.ts:59-98 implements exponential-backoff retry on 429/502/503 (3 retries, 1–8 s delay). Sustained outage manifests as a stuck run, surfaced by the supervisor — operator falls back to running Cursor locally and replaying the workflow with the implementation step manually skipped.
Reviewer mis-calibration — trust score is a 0–100 LLM-emitted number; an over-confident reviewer recommends APPROVE on a critical security bug. High Reviewer's guardrail sets requireHumanApproval: true and the merge gate is a separate workflow step — the reviewer never has merge authority. Production merges are policy-blocked until a human approves, regardless of trust score.
