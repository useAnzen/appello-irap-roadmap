1 · Objective

Design and ship the uis-decomposer workflow (8 steps, ~1.4 min avg duration) that takes an approved UIS vision and breaks it into dependency-ordered "hamburgers" — right-sized work packages that can each be independently implemented, tested, and merged by the Hamburger Factory (C1-0003). Each hamburger carries parent-vision references, scoped user stories and ACs, technical spec slice, and predecessor/successor expectations. The agent enforces decomposition principles (data model → API → UI, list before detail, CRUD order) and sizing rules (≤5 stories, ≤15 ACs, ≥2 ACs per hamburger).

Current production reality: success rate is 22% (9 runs to date — see C1-0003 §4). This epic ships the workflow + agent contract; the eval-driven prompt iteration to raise success rate is the next-phase work scored against C1-0034 golden manifests.

2 · Acceptance Criteria

  • 8-step workflow registered as uis-decomposer (workflow ID cmmxlv4qa00at8e12h8rdnp2a, version 3) in AgentC2.
  • Input schema accepts uisId, featureName, repository, uisOutput (full structured output from UIS Generator), plus optional slug and specUrl.
  • Step 1 (analyze-scope) returns hamburgers with sequenceNumber, title, description, scope, userStories, acceptanceCriteria, techSpec, predecessors, successors, successorExpectations plus a dependencyGraph and coverageReport.
  • Step 2 (validate-decomposition) runs the 7 validation rules below; returns { valid, issues, suggestions, coverageGaps }.
  • Human Gate 3 (decomposition-review-gate) integrates with Command Center; supports approved/feedback/reorder decisions.
  • Step 3 (assemble-manifest) titles each hamburger as [UIS-XX H1/N] Title and attaches parent-vision summary + full interface inventory + data-model decision.
  • Step 4 (write-children-to-supabase) creates child work_packages rows linked to the parent via parent_work_package_id using the write-hamburger-children tool.
  • Human Gate 4 (dispatch-gate) supports auto-approval thresholds: autoApprovePlanBelow + autoApprovePrBelow ∈ {low, medium, high}.
  • Steps 5–6 prepare and dispatch the Hamburger Factory input with workPackageId, hasUiInterfaces flag, and the full changeSurface.

5 · Decomposition Principles (Agent Contract)

  1. Data model first — Prisma schema changes and migrations come before any API or UI work.
  2. API before UI — GraphQL resolvers, queries, and mutations are built before the frontend consumes them.
  3. List before detail — list pages are implemented before detail/edit views.
  4. CRUD order — Create → Read → Update → Delete.

Hamburger Sizing Rules

Max user stories per hamburger5
Max acceptance criteria per hamburger15

6 · Step Reference

Step 1 — analyze-scope (agent: uis-decomposer, Claude Sonnet, JSON output)

Breaks the UIS into hamburgers following the principles above. Output:

{
  hamburgers: [
    {
      sequenceNumber: 1,
      title: "Data Model + Prisma Schema",
      description: "...",
      scope: { ... },
      userStories: [ ... ],
      acceptanceCriteria: [ ... ],
      techSpec: { ... },
      predecessors: [],
      successors: [2, 3],
      successorExpectations: [ ... ]
    },
    ...
  ],
  dependencyGraph: { ... },
  coverageReport: { ... }
}

Step 2 — validate-decomposition (agent: uis-decomposer, Claude Sonnet)

Second invocation runs 7 checks:

  1. Every user story maps to exactly one hamburger (no gaps, no duplicates)
  2. Every AC maps to exactly one hamburger
  3. Dependency graph is acyclic (topologically sortable)
  4. No hamburger exceeds size limits (>5 stories or >15 ACs)
  5. No hamburger is too small (<2 ACs — should be merged)
  6. Predecessor/successor expectations are consistent across hamburgers
  7. Sequencing follows the decomposition principles

Output: { valid, issues[], suggestions[], coverageGaps: { uncoveredStories[], uncoveredACs[] } }.

Gate 3 — decomposition-review-gate (HUMAN)

Form fields: approved (boolean), feedback (string), reorder (boolean). Reviewer sees hamburger breakdown, dependency graph, coverage report, and validation result.

Step 3 — assemble-manifest (transform)

Builds the ordered manifest. Each hamburger entry includes sequenceNumber, prefixed title ([UIS-XX H1/N] Title), description, repository, uisId, specUrl, scoped subset of scope/stories/ACs/techSpec, full codebaseAnalysis, full interfaceInventory, dataModelDecision, parent-vision summary, predecessors, successors, and successorExpectations.

Step 4 — write-children-to-supabase (tool: write-hamburger-children)

Creates child work_packages rows in the appello-sdlc Supabase DB (see C1-0001). Each hamburger becomes a child of the parent UIS work package via parent_work_package_id. Input: parentSlug, uisId, hamburgers. Output: { childrenCreated, children[]: { slug, workPackageId, sequenceNumber } }.

Gate 4 — dispatch-gate (HUMAN)

Form fields: dispatch (boolean), autoApprovePlanBelow (low/medium/high), autoApprovePrBelow (low/medium/high). Thresholds gate auto-approval at the per-hamburger level inside the Factory.

Step 5 — prepare-factory-input (transform)

Merges manifest data with Supabase-generated slugs and work package IDs. Each factory-bound hamburger carries sequenceNumber, slug, workPackageId, title, description, predecessors, successors, scope, acceptanceCriteria, changeSurface, and hasUiInterfaces (drives whether E2E tests run post-merge).

Step 6 — dispatch-hamburger-factory

Invokes the hamburger-factory workflow with the prepared manifest.

7 · Implementation Phases (36h)

  1. Workflow + step skeleton (6h) — register the 8-step uis-decomposer workflow shell; wire input schema and gate definitions.
  2. Decomposer agent prompt + JSON contract (10h) — author the system prompt that bakes in the 4 principles and 5 sizing rules; define the strict JSON output schema.
  3. Validation agent prompt (6h) — second invocation with the 7 validation rules and coverage-gap reporting.
  4. Manifest assembly + Supabase write (6h) — transform step + integration with the write-hamburger-children tool against the C1-0001 schema.
  5. Human gates + Command Center integration (4h) — wire Gate 3 and Gate 4 to workflow_resume and the auto-approve threshold logic.
  6. Factory dispatch + smoke run (4h) — Step 5–6 end-to-end against a real UIS, end-to-end run, persisted to Supabase.

8 · Risks & Mitigations

  • Low success rate (22% today) — the agent over-merges or under-covers. Mitigation: golden manifest benchmark set (C1-0034) drives prompt iteration; per-rule scoring isolates which validation check is failing.
  • Coverage drift on long UISs — large feature specs cause ACs to be silently dropped. Mitigation: Step 2 validation hard-fails on any uncovered story/AC; gate cannot be reached with coverage gaps.
  • Dependency cycles from LLM — agent occasionally proposes mutual predecessors. Mitigation: topological sort check in Step 2; cycles trigger automatic re-decompose with feedback.
  • Auto-approve overreach — thresholds set too permissive could ship bad PRs. Mitigation: default to no auto-approve; require explicit per-run threshold; trust score gating in C1-0053.
  • Schema drift in write-hamburger-children — tool input schema diverges from C1-0001 Prisma. Mitigation: shared TypeScript types generated from Prisma; tool contract test in CI.
