1 · Objective

Stand up the production runtime that hosts every other layer of the AgentC2 platform — the hamburger factory (SDLC autonomous build pipeline) and the Cursor Cloud dispatch path that lets agents spawn isolated cloud sessions. The runtime is three managed surfaces glued together by GitHub Actions:

  • Supabase Postgres as the durable system of record (with pgvector for the RAG index, see C1-0001).
  • Inngest as the durable job queue / function runtime that fans out work — goal submissions, evaluations, learning sessions, IFC parsing, simulations.
  • DigitalOcean droplet running the three-container Docker Compose stack (frontend / agent / admin) behind Caddy with health-checked rolling restarts.

Everything is provisioned through committed config (docker-compose.production.yml, .github/workflows/deploy-do.yml) with secrets centralised in Doppler. The deliverable for C1-0011 is the runtime itself plus the reproducible scripts and workflows that operate it.

2 · Acceptance Criteria

  1. Supabase project provisioned with pgvector extension enabled; DATABASE_URL wired into the agent / frontend / admin containers via Doppler.
  2. Inngest workers registered (project id mastra-agent) and syncing functions automatically on every deploy via PUT https://agentc2.ai/api/inngest.
  3. DigitalOcean droplet running docker compose -f docker-compose.production.yml up -d with all three services (agentc2-frontend, agentc2-agent, agentc2-admin) reporting healthy.
  4. CI/CD green: deploy-do.yml runs on every push to main (production) and develop (staging) with concurrency-gated, zero-downtime rolling deploy and Slack notifications on success / failure.
  5. Security gates (security-gates.yml) run on every PR and nightly cron: type-check, lint, tests, bun audit, audit-ci --high, license check, SBOM (CycloneDX), and Gitleaks secret scan.
  6. All secrets in Doppler under project agentc2 with three configs (dev / staging / production) and a CI service token wired to GitHub Actions.
  7. Caddy reverse proxy serving https://agentc2.ai with automatic TLS; /, /login, and /admin all return 200 in the post-deploy health check.
  8. Docker isolation: failed builds never take down running containers (verified in the deploy script's build → up sequence).

3 · Runtime topology

The production runtime is a hub-and-spoke layout. GitHub Actions is the single deploy entry point; the droplet runs every long-lived process; Supabase + Inngest are the two managed dependencies; Cursor Cloud dispatches into the agent container via the Docker socket mount.

The agent container is the only service that mounts /var/run/docker.sock — that mount is what gives Cursor Cloud dispatch the ability to spawn sibling containers on the same host.

4 · Docker Compose — production services

Source: docker-compose.production.yml

Three first-party services share a single .env file (populated by Doppler at deploy time), each with its own healthcheck, memory limit, and exposed port. No bind-mount Caddy here — Caddy runs on the host as a systemd service and reverse-proxies to localhost ports.

4.1 Frontend — public app on :3000, 1G cap

services:
  frontend:
    build:
      context: .
      dockerfile: apps/frontend/Dockerfile
      args:
        NEXT_PUBLIC_APP_URL: ${NEXT_PUBLIC_APP_URL:-https://agentc2.ai}
    container_name: agentc2-frontend
    ports: ["3000:3000"]
    env_file: .env
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:3000/"]
      interval: 30s
      timeout: 5s
      start_period: 10s
      retries: 3
    deploy:
      resources:
        limits:
          memory: 1G

4.2 Agent — hamburger factory + Cursor dispatch, :3001, 6G cap, Docker socket mounted

  agent:
    build:
      context: .
      dockerfile: apps/agent/Dockerfile
    container_name: agentc2-agent
    ports: ["3001:3001"]
    env_file: .env
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock   # Cursor dispatch spawns sibling containers
      - /var/lib/agentc2:/var/lib/agentc2             # shared work volume (factory artifacts)
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:3001/"]
      interval: 30s
      timeout: 10s
      start_period: 60s    # agent boots slowly — give Inngest sync time
      retries: 3
    deploy:
      resources:
        limits:
          memory: 6G

4.3 Admin — internal console on :3003, 512M cap

  admin:
    build:
      context: .
      dockerfile: apps/admin/Dockerfile
    container_name: agentc2-admin
    ports: ["3003:3003"]
    env_file: .env
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:3003/admin"]
      interval: 30s

Networking model. Compose's default bridge network is used — services discover each other by container name (agentc2-frontend / agentc2-agent / agentc2-admin) and Caddy on the host routes agentc2.ai → localhost:3000, agentc2.ai/admin → localhost:3003, etc. The legacy docker-compose.yml (local dev) defines a separate catalyst-network bridge for the MySQL dev DB; production never uses it.

5 · CI/CD pipeline — deploy-do.yml

Source: .github/workflows/deploy-do.yml

Single workflow, two jobs (test → deploy), 20-minute test timeout, 45-minute deploy timeout. Triggers on push to main (production) or develop (staging), plus manual workflow_dispatch with a skip_tests escape hatch.

5.1 Trigger + concurrency — one deploy at a time per environment

on:
  push:
    branches: [main, develop]
  workflow_dispatch:
    inputs:
      skip_tests:    { type: boolean, default: "false" }
      environment:   { type: choice, options: [production, staging] }

concurrency:
  group: deploy-${{ github.ref_name == 'develop' && 'staging' || 'production' }}
  cancel-in-progress: true

env:
  DEPLOY_PATH: /var/www/agentc2
  NODE_OPTIONS: "--max-old-space-size=6144"

5.2 Quality gates (job test) — Bun 1.3.4, Prisma generate, turbo type-check + lint

      - name: Setup Bun
        uses: oven-sh/setup-bun@v2
        with: { bun-version: "1.3.4" }

      - name: Install dependencies
        run: bun install

      - name: Generate Prisma client
        run: cd packages/database && bun run --bun prisma generate
        env: { DATABASE_URL: "postgresql://placeholder:placeholder@localhost:5432/placeholder" }

      - name: Type check
        run: bunx turbo type-check --concurrency=1

      - name: Lint
        run: bunx turbo lint --concurrency=1

5.3 Deploy (job deploy) — SSH to droplet, git reset, compose build + up, prune

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ (github.ref_name == 'develop' ...) && secrets.DO_STAGING_HOST || secrets.DO_HOST }}
          username: ${{ secrets.DO_USERNAME }}
          key: ${{ secrets.DO_SSH_KEY }}
          script_stop: true
          command_timeout: 35m
          script: |
            set -euo pipefail
            cd "$DEPLOY_PATH"

            # 1. Pull latest code
            git fetch origin
            git reset --hard origin/main

            # 2. Reload Caddy config (host-level, not containerised)
            sudo cp apps/caddy/Caddyfile.production /etc/caddy/Caddyfile
            sudo systemctl reload caddy || sudo systemctl restart caddy

            # 3. Build new images — old containers keep running (isolation!)
            docker compose -f docker-compose.production.yml build

            # 4. Rolling swap to new containers
            docker compose -f docker-compose.production.yml up -d

            # 5. Verify + prune
            sleep 10
            docker compose -f docker-compose.production.yml ps
            docker image prune -f

5.4 Post-deploy — health check, Inngest sync, Slack notify

      - name: Health check
        run: |
          sleep 30
          HTTP_HOME=$(curl -sf -o /dev/null -w "%{http_code}" https://agentc2.ai/ || echo "000")
          HTTP_LOGIN=$(curl -sf -o /dev/null -w "%{http_code}" https://agentc2.ai/login || echo "000")
          HTTP_ADMIN=$(curl -sf -o /dev/null -w "%{http_code}" https://agentc2.ai/admin || echo "000")
          if [ "$HTTP_HOME" = "000" ] || ... ; then
            echo "::warning::Health check could not reach one or more production URLs"
          fi

      - name: Sync Inngest functions
        run: |
          curl -sf -X PUT https://agentc2.ai/api/inngest

      - name: Notify Slack (success / failure)
        # posts to SLACK_WEBHOOK_URL with run link + commit message
Zero-downtime guarantee. Steps 3 and 4 are the load-bearing pair — docker compose build creates new images out-of-band, so a broken build leaves the current containers untouched. Only up -d swaps to the new images, and Compose handles the rolling restart per service.

6 · Security gates — security-gates.yml

Source: .github/workflows/security-gates.yml

Runs on every PR and nightly at 06:00 UTC. Same Bun 1.3.4 + Prisma generate + turbo type-check + lint baseline as the deploy workflow, then layers on tests, SCA, license, SBOM, and secret scan. Continues on failure for the soft gates (|| true) so the SBOM still uploads even if a downstream check fails.

6.1 Triggers + base setup

on:
  pull_request:
  schedule:
    - cron: "0 6 * * *"

jobs:
  security:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
        with: { bun-version: "1.3.4" }
      - run: bun install --frozen-lockfile
      - run: cd packages/database && bun run --bun prisma generate

6.2 Static gates — type-check, lint, unit + integration tests

      - name: Type check
        run: bunx turbo type-check --concurrency=1
      - name: Lint
        run: bunx turbo lint --concurrency=1
      - name: Run unit and integration tests
        run: bun run test

6.3 Supply-chain gates — audit, SCA, license, SBOM, secrets

      - name: Dependency audit
        run: bun audit || true

      - name: SCA scan (audit-ci)
        run: bunx audit-ci --high || true

      - name: License compliance check
        run: |
          bunx license-checker --production \
            --failOn "GPL-2.0;GPL-3.0;AGPL-1.0;AGPL-3.0" --summary || true

      - name: Generate SBOM (CycloneDX)
        run: bunx @cyclonedx/cyclonedx-npm --output-file sbom.json --spec-version 1.5 || true

      - name: Upload SBOM artifact
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.json
          retention-days: 90

      - name: Secret scan
        uses: gitleaks/gitleaks-action@v2
        continue-on-error: true
        env: { GITLEAKS_LICENSE: ${{ secrets.GITLEAKS_LICENSE }} }

The 90-day SBOM retention plus the nightly cron mean we always have a recent, auditable bill of materials for the production runtime — the artefact that satisfies any IRAP / NRC supply-chain question.

7 · Inngest configuration — the durable job queue

Source: apps/agent/src/lib/inngest.ts

Single Inngest client (id mastra-agent) typed with EventSchemas.fromRecord so every event payload is strongly typed across the whole agent codebase. The full registry currently spans goal lifecycle, run evaluation, guardrails, budgets, closed-loop learning, daily metrics rollups, simulations, and BIM/IFC parsing — every async surface in the platform.

import { Inngest, EventSchemas } from "inngest";

export const inngest = new Inngest({
    id: "mastra-agent",
    schemas: new EventSchemas().fromRecord<{
        // Goal lifecycle
        "goal/submitted":        { data: { goalId: string; userId: string } };
        "goal/retry":            { data: { goalId: string; userId: string; attempt: number } };

        // Agent workspace
        "run/completed":         { data: { runId: string; agentId: string;
                                            status: "COMPLETED" | "FAILED";
                                            durationMs?: number; totalTokens?: number;
                                            costUsd?: number; organizationId?: string; ... } };
        "run/evaluate":          { data: { runId: string; agentId: string; ... } };
        "evaluation/completed":   { data: { evaluationId: string; runId: string;
                                            scores: Record<string, number>; ... } };
        "guardrail/event":       { data: { type: "BLOCKED"|"MODIFIED"|"FLAGGED";
                                            guardrailKey: string; reason: string; ... } };
        "budget/check":           { data: { agentId: string } };
        "budget/alert":           { data: { agentId: string; currentUsage: number;
                                            limit: number; percentUsed: number } };

        // Closed-loop learning
        "learning/signal.detected":    { data: { agentId: string; signalType: string; ... } };
        "learning/session.start":      { data: { agentId: string;
                                                    triggerType?: "threshold"|"scheduled"|"manual"; ... } };
        "learning/session.scheduled":  { data: { timestamp?: string } };
        "learning/signals.extract":    { data: { sessionId; agentId; datasetId } };
        "learning/proposals.generate": { data: { sessionId; agentId; signalCount } };
        "learning/experiment.run":     { data: { sessionId; experimentId; agentId } };
        "learning/approval.request":   { data: { sessionId; agentId; proposalId } };
        "learning/version.promote":    { data: { sessionId; approvedBy: string; ... } };

        // Daily rollups + simulation + BIM (abridged)
        "metrics/daily.rollup":        { data: { date?: string } };
        "simulation/session.start":    { data: { sessionId; agentId; theme: string;
                                                    targetCount: number; concurrency: number } };
        "simulation/batch.run":        { data: { sessionId; agentId; theme: string;
                                                    batchIndex: number; batchSize: number } };
        "bim/ifc.parse":               { data: { modelId; versionId;
                                                    sourceKey: string; sourceUri?: string } };
    }>()
});
Sync mechanism. The agent container exposes /api/inngest; the deploy workflow's "Sync Inngest functions" step (deploy-do.yml:169-173) hits PUT https://agentc2.ai/api/inngest after every deploy so Inngest Cloud always knows the latest function signatures. No manual sync needed.

8 · Secrets & IAM — Doppler

Source: scripts/setup-doppler.sh

All runtime secrets — Supabase DATABASE_URL, Inngest signing key, OpenAI / Anthropic keys, Cursor Cloud dispatch token, Slack webhook — live in Doppler under a single project agentc2 with three configs: dev, staging, production. The deploy droplet writes .env from Doppler before each docker compose up, and the GitHub Actions workflow consumes a CI service token.

8.3 IAM surface

  • GitHub Actions → DigitalOcean: DO_HOST / DO_USERNAME / DO_SSH_KEY (production) and DO_STAGING_HOST (staging) live as repo secrets; no IAM role on the cloud side — pure SSH.
  • Droplet → Supabase: connection string from Doppler, RLS enforced inside Postgres (see C1-0001 for the per-tenant policy set).
  • Agent → Docker socket: scoped by file-system permission on /var/run/docker.sock; only the agent container mounts it.
  • CI service token: DOPPLER_TOKEN is config-scoped to production and stored only in GitHub Actions secrets (never committed, never on the droplet's disk).

10 · Risks & mitigations

RiskLikelihoodImpactMitigation
Cost overrun — agent container has a 6G memory ceiling and runs Cursor Cloud dispatch; concurrent runs can saturate the droplet and force a vertical resize, OR Inngest function-execution + Supabase egress can spike with the learning + simulation event volume. Medium Medium Per-agent budget events (budget/check, budget/alert) already wired in inngest.ts; the daily metrics rollup (metrics/daily.rollup) lets us trend cost. Add a DigitalOcean billing alert at 80% of the monthly cap and a Supabase usage alert at 70%.
Secret rotation drift — Doppler is the source of truth, but a manually-edited .env on the droplet (or a service token left valid past its rotation window) silently diverges and creates a hard-to-diagnose auth break. Medium High Forbid manual .env edits on the droplet — deploy script always re-sources from Doppler. Calendar reminders for the 90 / 180 / 365-day cadences from setup-doppler.sh:72-75. Gitleaks in nightly cron catches accidental commits.
Single-region SPOF — one droplet, one Caddy, one Supabase region. A regional DigitalOcean outage takes the entire factory down and the deploy workflow has no fallback host. Low High Staging droplet (DO_STAGING_HOST) lives in a different region and runs the same image; promote staging → production via DNS swap as the emergency runbook. Long-term: warm standby droplet driven by a second branch of the deploy workflow.
Docker socket mount blast radius — the agent container has root-equivalent host access through /var/run/docker.sock. A compromised agent process can spawn / kill any container on the host. Low High Treat the agent image as security-critical; security-gates SBOM + audit-ci run on every PR. Long-term migration to a dedicated dispatch sidecar with a scoped Docker API proxy (filed as C1 follow-up).
