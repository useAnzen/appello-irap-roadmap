1 · Objective

Stand up a first-class local-development experience for the AgentC2 monorepo so a brand-new engineer can clone the repo and run the full SDLC pipeline — frontend, agent app, admin, Inngest background jobs, Caddy HTTPS proxy, Prisma against Postgres — in under 30 minutes from a clean macOS workstation. The repo is a Bun-managed Turborepo with workspaces under apps/* and packages/*; this epic delivers the workspace topology, the Turbo task graph, the docker-compose surface, the curated seed-script library, and the README onboarding script that ties them together.

The activity is a foundation for everything downstream in C1: the schema work (C1-0001), the pipeline architecture (C1-0003), and the production infra mirror (C1-0011) all assume a developer can boot the same stack locally. Without a clean bootstrap, every contributor builds their own snowflake environment and the flywheel stalls.

2 · Acceptance Criteria

  1. Bun + Turborepo workspace healthy — root package.json declares "packageManager": "bun@1.3.4" and "workspaces": ["apps/*", "packages/*"]; bun install resolves every workspace:* link without error.
  2. turbo run dev is the single entrypoint — one command boots Caddy, frontend (3000), agent (3001), admin (3003), and Inngest (8288) concurrently via the Turbo TUI. bun run dev:local bypasses Caddy for engineers who do not want HTTPS.
  3. docker compose up -d brings the local data plane up — at least one database container starts and survives a healthcheck loop. See §7 — the shipped docker-compose.yml spins up MySQL only; the Postgres expected by the app is sourced from a managed DB (Supabase / DO) per .env.example. This deviation from the original brief is documented in §13.
  4. Seed scripts populate fixtures — bun run db:seed hits the Turbo db:seed task; bun run db:seed-admin and bun run db:seed-test-user drop a working admin user and an E2E test user with Better Auth scrypt-hashed credentials.
  5. README onboarding ≤ 30 minutes — a 9-step quickstart in README.md takes a new engineer from git clone to https://catalyst.localhost; every step is copy-pasteable and idempotent.
  6. All turbo run commands green on a fresh clone — build, lint, type-check, db:generate all succeed; the Turbo cache is populated and a second run is a no-op.
  7. Destructive-change guardrail — bun run db:push routes through scripts/safe-db-push.sh which aborts if Prisma would drop an externally-managed table (e.g. rag_documents from @mastra/pg).
  8. Production parity — docker-compose.production.yml mirrors the same service topology (frontend / agent / admin) for the prod deploy path, sharing build args and env_file conventions with local dev. See C1-0011 for the prod-infra side.

4 · Turborepo configuration (real turbo.json)

Single root turbo.json defines the task graph for every workspace. Key invariants: build depends on upstream ^build and ^db:generate so Prisma client generation always runs before any app builds; dev is non-cached and persistent; db:* tasks are uncached because they have external side effects.

4.2 turbo.json task graph

Source: turbo.json (verbatim)

{
    "$schema": "https://turbo.build/schema.json",
    "globalEnv": ["NEXT_PUBLIC_APP_URL"],
    "ui": "tui",
    "tasks": {
        "build": {
            "dependsOn": ["^build", "^db:generate"],
            "inputs": ["src/**", "app/**", "*.ts", "*.tsx", "*.mjs", "*.json",
                       "!**/*.test.*", "!**/*.spec.*", "!**/__tests__/**"],
            "outputs": [".next/**", "!.next/cache/**", "dist/**"]
        },
        "dev": {
            "dependsOn": ["^db:generate"],
            "cache": false,
            "persistent": true
        },
        "lint":        { "dependsOn": ["^build", "^db:generate"], "inputs": ["src/**", "app/**", "*.ts", "*.tsx", "*.mjs", "*.json", "eslint.config.*"] },
        "type-check":  { "dependsOn": ["^build", "^db:generate"], "inputs": ["src/**", "app/**", "*.ts", "*.tsx", "*.mjs", "*.json", "tsconfig.json"] },
        "test":        { "dependsOn": ["^build", "^db:generate"], "cache": false },
        "clean":       { "cache": false },
        "db:generate": { "inputs": ["prisma/schema.prisma", "prisma/migrations/**"], "outputs": [] },
        "db:push":     { "cache": false },
        "db:migrate":  { "cache": false },
        "db:studio":   { "cache": false },
        "db:seed":     { "cache": false },
        "storybook":   { "cache": false, "persistent": true }
    }
}

4.3 Why this shape

  • ^build + ^db:generate on every code task — guarantees the Prisma client is generated before any TypeScript compiles against it. Removes the classic monorepo "@prisma/client is undefined" bootstrap order bug.
  • dev is persistent: true — Turbo TUI keeps the per-app log panes alive and never tries to cache a long-running dev server.
  • "ui": "tui" — opt into the new Turborepo terminal UI; each app gets a scrollable log pane in the TUI rather than interleaved stdout.
  • Per-package db:* scripts live in packages/database/package.json and route through dotenv -e ../../.env so they always read the repo-root env file even when invoked from a sub-workspace.

Deviation from the original brief — documented honestly. The shipped docker-compose.yml only declares a MySQL service (named catalyst-mysql on the catalyst-network), but the app's data layer is PostgreSQL per .env.example (DATABASE_URL="postgresql://…") and Prisma. In practice, engineers point DATABASE_URL at a managed Postgres (Supabase or DO Managed Postgres — see §6) instead of a containerized local Postgres. The MySQL container appears to be a legacy artifact from the predecessor "Catalyst" project that has not yet been removed. Recorded as a known gap in §13 (Risks) with the recommended follow-up: replace this file with a Postgres + pgvector service to match .env.example and the production stack. Until then, the README's step 5 (docker compose up -d) is a no-op for Postgres-only setups; engineers using managed Postgres can skip it.

  • Three services, same env_file — .env at the repo root is the single source of truth for both local and prod containers.
  • agent has docker.sock + /var/lib/agentc2 volume — required for agent-driven sandbox container spawning and persistent agent state.
  • Memory limits codify the cost model — agent (6G) ≫ frontend (1G) ≫ admin (512M).
  • basePath alignment — admin healthcheck hits /admin because the app sits behind a path-based reverse proxy (Caddy locally, the production reverse proxy in deployment).

7.2 Representative seed — seed-test-user.ts (the E2E fixture)

Source: scripts/seed-test-user.ts:1-60 (abridged)

#!/usr/bin/env bun
import { prisma } from "../packages/database/src";

async function hashPassword(password: string): Promise<string> {
    const { scrypt, randomBytes } = await import("node:crypto");
    const { promisify } = await import("node:util");
    const scryptAsync = promisify(scrypt);

    const salt = randomBytes(16).toString("hex");
    const derivedKey = (await scryptAsync(password.normalize("NFKC"), salt, 64, {
        N: 16384, r: 16, p: 1,
        maxmem: 128 * 16384 * 16 * 2
    })) as Buffer;
    return `${salt}:${derivedKey.toString("hex")}`;
}

async function main() {
    const email = process.env.TEST_USER_EMAIL || "e2e-test@catalyst.local";
    const password = process.env.TEST_USER_PASSWORD || "E2ETestPassword123!";

    const hashedPassword = await hashPassword(password);

    const user = await prisma.user.upsert({
        where:  { email },
        update: { name: "E2E Test User", emailVerified: true },
        create: { name: "E2E Test User", email, emailVerified: true }
    });
    // …upserts credential account, adds membership to every org…
}

8 · Developer onboarding — the README quickstart, verbatim

Source: README.md "Getting Started" (steps 1–9, real commands)

Step 1 · Clone the repository

git clone <repository-url>
cd agentc2

Step 2 · Install Caddy (local HTTPS)

brew install caddy
caddy trust    # installs root CA into the macOS keychain

Caddy enables the cookie-shared catalyst.localhost URL structure. Engineers who do not want HTTPS can skip this step and use bun run dev:local instead (see step 8 alternate).

Step 3 · Install dependencies

bun install

Bun resolves every workspace:* link across the 6 apps and 9 packages from a single root bun.lock.

Step 4 · Configure environment

cp .env.example .env
# edit .env: at minimum set DATABASE_URL + OPENAI_API_KEY + BETTER_AUTH_SECRET

Step 5 · Start the local data plane

docker compose up -d

See §7 — the shipped docker-compose.yml only spins up MySQL today. Engineers using a managed Postgres (Supabase / DO) can skip this step entirely and the rest of the bootstrap still works. Tracked as a follow-up to add a Postgres + pgvector service to match the prod stack.

Step 6 · Generate the Prisma client and push the schema

bun run db:generate
bun run db:push    # routes through scripts/safe-db-push.sh — aborts on destructive diffs

Step 7 · Seed the database (optional but recommended for new hires)

bun run db:seed                # runs every workspace's db:seed task via turbo
bun run db:seed-admin          # adds an admin user to the platform org
bun run db:seed-test-user      # E2E test user (e2e-test@catalyst.local)

Step 8 · Start the dev servers

bun run dev                    # Turbo TUI: Caddy + frontend + agent + admin + inngest
# or, without Caddy / HTTPS:
bun run dev:local              # turbo dev --filter=!caddy

Step 9 · Open the app

https://catalyst.localhost          # frontend (with shared cookies)
https://catalyst.localhost/agent    # agent app
https://catalyst.localhost/admin    # admin dashboard
http://localhost:8288               # Inngest dev dashboard
Gate ➔ The Caddy TUI pane shows green; the agent app loads; the test user can sign in.
Wall-clock target. On a clean macOS workstation with Homebrew + Docker Desktop already installed, an engineer should hit the gate at step 9 in ≤ 30 minutes — most of that is bun install (~2 min cold) and editing the half-dozen .env keys. Caddy install + trust is ~30 seconds. The first bun run dev is the longest single wait (~60s for all Next.js apps to compile their first route).

10.2 Seed rollback / idempotency

  1. Drop the database, run db:push + every seed script in sequence — capture SELECT count(*) across the populated tables.
  2. Re-run every seed unchanged. Assert deltas are zero (or are explicitly versioned, e.g. a new PlaybookVersion row for seed-platform-org.ts).
  3. Test the destructive-change guardrail: stage a schema change that would drop rag_documents, run bun run db:push, assert the banner fires and the schema is not modified.

11 · Risks & mitigations

RiskImpactMitigation
Bun version drift — engineers on Bun 1.2.x or 1.4.x see subtle workspace-resolution differences. The root declares "packageManager": "bun@1.3.4" but does not enforce it. Med The README "Prerequisites" block pins Bun ≥ 1.3.4. A future scripts/check-prereqs.sh hook from predev should validate Bun's version and fail-fast with a remediation message. asdf-style version files (.tool-versions / .bun-version) would harden this further.
Docker version compatibility — docker-compose.production.yml uses the deploy.resources.limits stanza which is silently ignored by Compose v1. Older engineering laptops may still have docker-compose (v1) instead of docker compose (v2). Med README steps use docker compose (v2 plugin syntax). Document the v2 requirement explicitly in Prerequisites. CI uses Docker 24+ on the runner; the prod deploy script (scripts/deploy-do.sh) shells out via docker compose.
Secret onboarding for new engineers — .env.example has ~50 keys; many are for optional integrations. New hires waste time provisioning OAuth apps they will not touch. Med The example file is grouped by concern; the README "minimum required" list is just DATABASE_URL + OPENAI_API_KEY + BETTER_AUTH_SECRET + CREDENTIAL_ENCRYPTION_KEY + NEXT_PUBLIC_APP_URL. Optional groups are explicitly labelled. Long-term mitigation: move shared secrets into Doppler (scripts/setup-doppler.sh is present in the repo as a stub) so the platform org provisions them centrally.
docker-compose.yml is MySQL-only but app wants Postgres — README step 5 is misleading; docker compose up -d starts catalyst-mysql which the app never reads. High Document the deviation in §5 + §13. Recommended follow-up: replace the MySQL service with a postgres:16 + pgvector bundle so docker compose up matches what .env.example implies. Until then, the README directs engineers to a managed Postgres (Supabase / DO Managed Postgres) and the local docker compose step is effectively optional.
Caddy root-CA trust on locked-down machines — caddy trust requires admin/sudo; some corporate Macs block it. Low bun run dev:local (Turbo filter --filter=!caddy) bypasses Caddy entirely; update .env to NEXT_PUBLIC_APP_URL=http://localhost:3000. Documented as the troubleshooting "Want to bypass Caddy?" step in the README.
Inngest dev server discovery races — the agent app must be up before Inngest can discover /api/inngest. A cold bun run dev sometimes races. Low scripts/start-inngest.sh defaults INNGEST_APP_URL=http://localhost:3001/api/inngest; npx inngest-cli@latest dev -u $URL retries on connection refused, so the race resolves on its own within a few seconds. No-op on subsequent restarts because the script early-exits when the port is already healthy.
