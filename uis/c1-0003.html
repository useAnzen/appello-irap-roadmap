1 · Objective

Architect the closed-loop AI development pipeline that turns a feature idea (or a customer bug ticket) into a merged pull request in useAnzen/application-mono-repo. The pipeline is composed of 6 active workflows orchestrated through Supabase-persisted work items, dispatched against Cursor Cloud coding agents, gated by GitHub CI, and supervised by 4 human decision gates at the highest-leverage junctures. This epic delivers the architecture: the workflow contracts, the agent roster, the gate model, the persistence schema integration with C1-0001, and the 5 reference diagrams (data-model, decision-tree, state-machine, sequence, deployment).

2 · Acceptance Criteria

  • 6 SDLC workflows specified end-to-end with input/output contracts: uis-generator, uis-decomposer, hamburger-factory, sdlc-feature, sdlc-bugfix, sdlc-triage.
  • 12 SDLC agents specified with model selection, role, and step ownership (see Section 6).
  • 4 active human decision gates documented with step IDs, decision payloads, and Command Center integration (see Section 5).
  • 5 reference diagrams checked into agentc2/docs/sdlc/ covering data model, decision tree, state machine, sequence, and deployment.
  • Two entry paths supported: Feature Forge (UIS-first) and Ticket Triage (bug/feature classification).
  • External system contracts defined for Supabase, GitHub, Cursor Cloud, GitHub Pages, Google Drive, Fathom, and ATLAS.
  • v4 evolution: Hamburger Factory operates fully autonomously (gates 5–7 removed in favour of wait-for-checks against GitHub CI).

3 · Pipeline Architecture

Two entry paths converge on a single merged-PR outcome. Feature Forge runs the UIS Generator and then routes to either the Decomposer (for complex features) or directly to the SDLC Feature workflow. Ticket Triage classifies an incoming Jira/GitHub issue and routes it to the appropriate sub-workflow.

// Source: agentc2/docs/sdlc/00-overview.md (mermaid converted to ASCII)
Feature Forge Path:
    [Feature Idea] -> [UIS Generator] -> Gate1 (UIS Review)
                          |
                          +--Gate2 dispatch:decompose--> [UIS Decomposer] -> Gate3 -> Gate4 -> [Hamburger Factory] -+
                          |                                                                                          |
                          +--Gate2 dispatch:direct---------------------------------------------> [SDLC Feature] -----+--> [Merged PR]
                                                                                                                     |
Ticket Path:                                                                                                         |
    [Bug/Feature Ticket] -> [SDLC Triage] --classification:bug--> [SDLC Bugfix] ----------------------------------+--+
                                          \--classification:feat-> [SDLC Feature] --------------------------------+

All workflows persist run state to Supabase (xqyixujilhaozfvepbbd) via the schema from C1-0001. GitHub interaction is via the consolidated GitHub MCP. Coding agents are dispatched to Cursor Cloud. The UIS specs themselves are published to GitHub Pages on useAnzen/appello-sdlc.

5 · The 4 Active Human Decision Gates

Source: agentc2/docs/sdlc/07-human-gates.md. The v4 architecture moved from 7 human gates to 4: the Hamburger Factory now operates fully autonomously (plan review, PR review, and fix-PR review are replaced by GitHub CI gating via wait-for-checks).

Gate 1 — UIS Review

Workflow: uis-generator · Step ID: uis-review-gate · After: Phase 5c (all 8 sections written to Supabase)

Reviewer sees business-case, scope, ui-concepts, data-model, api-surface, change-surface, acceptance-criteria, dependencies. Decisions: {approved:true} / {rejected:true, feedback} / {sectionsToRevise:[...], feedback}.

Gate 2 — Dispatch Decision

Workflow: uis-generator · Step ID: sdlc-dispatch-gate · After: Phase 6b (UIS published to GitHub Pages)

Decisions: decompose (complex feature, route to UIS Decomposer) · direct (small self-contained, route to SDLC Feature) · skip (manual dispatch later).

Gate 3 — Decomposition Review

Workflow: uis-decomposer · Step ID: decomposition-review-gate

Reviewer approves or rejects the proposed hamburger breakdown and dependency graph. Coverage-validated against parent UIS (see C1-0029).

Gate 4 — Factory Dispatch

Workflow: uis-decomposer · Step ID: dispatch-gate

Final approval before hamburgers are written to Supabase as children and dispatched to the autonomous Hamburger Factory.

Deprecated (v4): Gates 5–7

Gate 5 — Plan Review

Replaced by audit + CI gating per hamburger.

Gate 6 — PR Merge Review

Replaced by wait-for-checks on GitHub CI + trust-score threshold (see C1-0053).

Gate 7 — Fix PR Review

Replaced by Self-Healing Cycle running autonomously.

8 · Hamburger Factory Sub-Pipeline (v4, fully autonomous)

Per-hamburger execution inside the Factory after Gate 4:

// Source: agentc2/docs/sdlc/07-human-gates.md
[Plan + Audit] -> [Cursor Cloud Implementation] -> [GitHub CI: wait-for-checks] -> [Squash Merge] -> [Post-Merge CI]
                                                                                                          |
                                                                                                          +-- fail --> [Self-Healing Cycle (autonomous)]
                                                                                                          +-- pass --> done

The self-healing cycle is what enables fire-and-forget autonomy on bugfixes: a failed post-merge test triggers sdlc-healer to diagnose and open a fix PR, re-running the same CI gate. Loop terminates on green or after N attempts (N = heal-budget per hamburger).

9 · Implementation Phases (40h)

  1. Workflow contract design (12h) — author all 6 workflow step specs, input/output schemas, retry/escalation rules.
  2. Gate model design (8h) — define the 4 active gates and their decision payloads; spec the Command Center integration via workflow_resume MCP tool.
  3. Agent contract mapping (10h) — wire each of the 12 agents to the steps it owns; specify model selection, prompt skeleton, and I/O JSON schema.
  4. Diagram authoring (6h) — 5 mermaid diagrams (data-model, decision-tree, state-machine, sequence, deployment) checked into agentc2/docs/sdlc/.
  5. Cross-doc reconciliation + review (4h) — ensure overview, per-workflow docs, and gate doc agree.

10 · Risks & Mitigations

  • Agent-contract drift across deploys — model upgrades change behaviour. Mitigation: versioned playbook seed (C1-0015), agent eval suite per release.
  • Gate fatigue / human bottleneck — too many gates throttle throughput. Mitigation: v4 collapse from 7 to 4 gates; trust score auto-bypass below threshold (C1-0053).
  • Cursor Cloud SPOF — single vendor for coding agent dispatch. Mitigation: abstraction layer over coding-agent backends; fallback to local Cursor on outage.
  • Workflow versioning — schema/contract changes mid-run can corrupt persisted state. Mitigation: idempotent migrations via C1-0001 schema discipline; run-level contract pinning.
