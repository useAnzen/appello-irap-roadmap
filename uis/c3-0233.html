1 · Objective

agentc2/apps/agent/src/types/elevenlabs-websocket.ts is the existing WebSocket type pattern; agentc2/packages/agentc2/src/sessions/manager.ts handles open sessions; ChannelSession persists them. This epic builds a dashboard live agent-status panel that consumes those primitives — showing which agents are currently active, their queue depth, and current task — across the admin app. Off-theme note: this is closer to C1/C4 dashboard work but uses the same WebSocket infra as voice channels.

2 · Acceptance Criteria

  • Panel at /admin/agents/live shows real-time list of active agents.
  • Each row: agent slug, model, current task summary, runtime, queue depth.
  • Updates via WebSocket subscription to agents:live channel (multiplexed via existing sessions/manager.ts).
  • Channel events: agent.task.started, agent.task.progress, agent.task.completed, agent.queue.depth.
  • Disconnect-resilient: reconnect with exponential backoff; gap-fill via REST on reconnect.
  • Panel data persisted to ChannelSession so refresh shows the last-known state immediately.
  • Admin-only access via existing RBAC.
  • Soak: 50 concurrent admin viewers + 30 active agents holds steady < 100ms tick latency.

3 · Architecture & Approach

Channel layout

// Server: publishers per agent runtime
broadcaster.emit('agents:live', { type: 'agent.task.started', agentId, taskSummary, at });

// Client: single subscription, fan-out per agent row
const { events } = useChannel('agents:live');
useEffect(() => {
  events.on('agent.task.started', updateRow);
  // ...
}, []);

5 · Risks & Mitigations

  • WebSocket fan-out cost. Mitigation: shared broker process; per-pod connection cap.
  • Stale ChannelSession bloat. Mitigation: TTL 24h.
  • Off-theme creep into voice features. Mitigation: dedicated channel namespace; never reuse the elevenlabs one.
