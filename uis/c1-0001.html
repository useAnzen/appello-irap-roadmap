1 · Objective

Design and ship the relational schema that makes the Appello SDLC software factory database-driven — every artifact an AI agent produces (a stage output, a code review, a build run, a test run, a doc, a state transition) is a first-class row in Postgres with stable IDs, FKs, status enums, and audit timestamps. The schema is the contract every downstream system (the Next.js operator UI, the AgentC2 dispatcher, the workflow callback webhook, the human-in-the-loop reviewer screens) reads and writes through Prisma.

Concretely: produce a single schema.prisma in packages/database/prisma/ that models the lifecycle Project → WorkPackage → StageOutput → Review → WorkItem → BuildRun → TestRun → DocArtifact, wires Better Auth (User / Session / Account / Verification) alongside it, and emits an idempotent TransitionLog trail so every state change is queryable forever.

2 · Acceptance Criteria

  1. A single Prisma schema file is committed at packages/database/prisma/schema.prisma with the Postgres provider, the prisma-client-js generator, and uses DATABASE_URL for runtime / DATABASE_URL_MIGRATE for migrations.
  2. Better Auth models (User, Session, Account, Verification) are present and mapped to user / session / account / verification tables exactly as Better Auth expects.
  3. A Project table evolves the legacy project row to hold targetRepo, stagingUrl, productionUrl, and a JSON autoAdvancePolicy blob — the per-project router config.
  4. The eight software-factory tables exist with the names WorkPackage, StageOutput, Review, WorkItem, BuildRun, TestRun, TransitionLog, DocArtifact; every one uses gen_random_uuid() UUIDs, Timestamptz for audit columns, and onDelete: Cascade wherever a child row is meaningless without its parent.
  5. StageOutput enforces a composite uniqueness constraint (workPackageId, stage, version) so re-running a stage bumps the version explicitly instead of silently overwriting.
  6. Every cost/observability field the dispatcher needs is on the schema: agentc2RunId, durationMs, tokenCost (Decimal 10,4), healAttempts on StageOutput; the same trio plus trustScore, reviewScore, reviewNotes, securityFindings on BuildRun; testResults, screenshots, playwrightTrace, retryCount on TestRun.
  7. Lookup indexes exist for every query the operator UI runs: WorkPackage (projectId, stage, status), StageOutput (workPackageId), Review (workPackageId, stage), WorkItem (workPackageId), BuildRun (workItemId, status), TestRun (workItemId), TransitionLog (entity, entityId) + (createdAt), DocArtifact (workPackageId).
  8. TransitionLog captures the full xstate-style transition record — entity, entityId, fromState, event, guard?, toState, actions[], metadata? — so the audit trail is replayable.
  9. prisma db push succeeds against a fresh Supabase Postgres; prisma generate emits a TypeScript client consumed via packages/database/src/index.ts; db:seed and the dev check-* scripts run green end-to-end.
  10. The schema is consumed by the Next.js operator app at apps/frontend/ through server actions (lib/actions/*) and a small set of webhook routes (/api/workflow-callback, /api/sync-poller) — no other application code talks to Postgres directly.

3 · Architecture & approach

The stack is intentionally boring so the interesting work can live in the agent layer:

  • Postgres on Supabase, single database for both Better Auth tables and the SDLC factory tables.
  • Prisma 6 (@prisma/client ^6.2.1) as the single ORM. No raw SQL outside $executeRawUnsafe escape hatches.
  • Next.js (App Router) in apps/frontend/ consumes Prisma through packages/database/src/index.ts (re-exports prisma client + Prisma types). Server actions live in apps/frontend/src/lib/actions/.
  • Better Auth sits beside the factory tables in the same database; the only cross-link from factory → auth is Review.reviewerId → User.id so a human-reviewer verdict is attributable.
  • AgentC2 is the dispatcher — it owns workflow runs, calls back via /api/workflow-callback, and writes agentc2_run_id onto StageOutput / BuildRun / TestRun / DocArtifact so every artifact is traceable to the run that produced it.

4.2 Project — the root container, per-customer routing

Source: schema.prisma:73-89

model Project {
  id                String        @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  name              String
  slug              String        @unique
  description       String        @default("")
  targetRepo        String        @map("target_repo")
  stagingUrl        String?       @map("staging_url")
  productionUrl     String?       @map("production_url")
  autoAdvancePolicy Json          @default("{}") @map("auto_advance_policy")
  createdAt         DateTime      @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt         DateTime      @default(now()) @map("updated_at") @db.Timestamptz(6)
  workPackages      WorkPackage[]

  @@map("projects")
}

autoAdvancePolicy is a free-form JSON blob: which stages skip human review, which stages auto-dispatch the next, what trust-score threshold to advance at. Tested via packages/database/src/seed-projects.ts.

4.3 WorkPackage — the unit of agent work

Source: schema.prisma:93-114

model WorkPackage {
  id           String        @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  projectId    String        @map("project_id") @db.Uuid
  slug         String        @unique
  title        String
  description  String        @default("")
  stage        String        @default("draft")
  status       String        @default("draft")
  priority     Int           @default(0)
  createdAt    DateTime      @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt    DateTime      @default(now()) @map("updated_at") @db.Timestamptz(6)
  project      Project       @relation(fields: [projectId], references: [id])
  stageOutputs StageOutput[]
  reviews      Review[]
  workItems    WorkItem[]
  docArtifacts DocArtifact[]

  @@index([projectId])
  @@index([stage])
  @@index([status])
  @@map("work_packages_v2")
}

Note the table name work_packages_v2 — chosen deliberately because a legacy work_packages table predated this schema; the _v2 suffix lets the cutover run side-by-side without disturbing legacy readers. stage and status are both string-typed (open vocabulary), driven from the xstate machine that ships in C1-0002.

4.4 StageOutput — the durable artifact for each stage execution

Source: schema.prisma:116-135

model StageOutput {
  id            String      @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workPackageId String      @map("work_package_id") @db.Uuid
  stage         String
  version       Int         @default(1)
  content       String      @default("")
  format        String      @default("json")
  status        String      @default("generating")
  agentc2RunId  String?     @map("agentc2_run_id")
  durationMs    Int?        @map("duration_ms")
  tokenCost     Decimal?    @map("token_cost") @db.Decimal(10, 4)
  healAttempts  Int         @default(0) @map("heal_attempts")
  createdAt     DateTime    @default(now()) @map("created_at") @db.Timestamptz(6)
  workPackage   WorkPackage @relation(fields: [workPackageId], references: [id], onDelete: Cascade)
  reviews       Review[]

  @@unique([workPackageId, stage, version])
  @@index([workPackageId])
  @@map("stage_outputs")
}

The (workPackageId, stage, version) composite unique is the single most important constraint in the schema. Re-running a stage must create a new row with version + 1; silently mutating an existing row would destroy the audit trail and break the heal loop's "previous attempt's feedback" lookup. healAttempts is a counter for the auto-heal subsystem (C1-0003); when it exceeds the per-stage cap the dispatcher escalates to a human reviewer.

4.5 Review — verdict on a stage output (human or AI)

Source: schema.prisma:137-154

model Review {
  id             String       @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workPackageId  String       @map("work_package_id") @db.Uuid
  stageOutputId  String?      @map("stage_output_id") @db.Uuid
  stage          String
  verdict        String
  feedback       String?
  reviewerId     String?      @map("reviewer_id")
  triggeredRunId String?      @map("triggered_run_id")
  createdAt      DateTime     @default(now()) @map("created_at") @db.Timestamptz(6)
  workPackage    WorkPackage  @relation(fields: [workPackageId], references: [id], onDelete: Cascade)
  stageOutput    StageOutput? @relation(fields: [stageOutputId], references: [id])
  reviewer       User?        @relation(fields: [reviewerId], references: [id])

  @@index([workPackageId])
  @@index([stage])
  @@map("reviews")
}

reviewerId is nullable on purpose — when an AI reviewer (e.g. the lint stage) produces the verdict there is no User row to link to. triggeredRunId records the AgentC2 run that the verdict kicked off (typically the next stage's dispatch).

4.6 WorkItem — atomic ticket inside a WorkPackage

Source: schema.prisma:156-177

model WorkItem {
  id                 String      @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workPackageId      String      @map("work_package_id") @db.Uuid
  slug               String      @unique
  title              String
  description        String      @default("")
  sequenceNumber     Int         @map("sequence_number")
  scope              Json        @default("{}")
  acceptanceCriteria Json        @default("[]") @map("acceptance_criteria")
  techSpec           Json        @default("{}") @map("tech_spec")
  predecessors       Int[]       @default([])
  successors         Int[]       @default([])
  status             String      @default("pending")
  createdAt          DateTime    @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt          DateTime    @default(now()) @map("updated_at") @db.Timestamptz(6)
  workPackage        WorkPackage @relation(fields: [workPackageId], references: [id], onDelete: Cascade)
  buildRuns          BuildRun[]
  testRuns           TestRun[]

  @@index([workPackageId])
  @@map("work_items")
}

predecessors / successors are Int[] referencing other WorkItem.sequenceNumbers in the same package — a lightweight DAG so the dispatcher can parallel-fan-out and gate on prerequisites without a join table. scope / acceptanceCriteria / techSpec are JSON blobs populated by the scope-decomposition stage output.

4.7 BuildRun — one AgentC2 build attempt → one PR

Source: schema.prisma:179-203

model BuildRun {
  id               String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workItemId       String    @map("work_item_id") @db.Uuid
  agentc2RunId     String?   @map("agentc2_run_id")
  status           String    @default("queued")
  branchName       String?   @map("branch_name")
  prNumber         Int?      @map("pr_number")
  prUrl            String?   @map("pr_url")
  trustScore       Decimal?  @map("trust_score") @db.Decimal(4, 3)
  reviewScore      Decimal?  @map("review_score") @db.Decimal(4, 3)
  reviewNotes      Json      @default("[]") @map("review_notes")
  securityFindings Json      @default("[]") @map("security_findings")
  errorMessage     String?   @map("error_message")
  healAttempts     Int       @default(0) @map("heal_attempts")
  durationMs       Int?      @map("duration_ms")
  tokenCost        Decimal?  @map("token_cost") @db.Decimal(10, 4)
  dispatchedAt     DateTime  @default(now()) @map("dispatched_at") @db.Timestamptz(6)
  completedAt      DateTime? @map("completed_at") @db.Timestamptz(6)
  workItem         WorkItem  @relation(fields: [workItemId], references: [id], onDelete: Cascade)
  testRuns         TestRun[]

  @@index([workItemId])
  @@index([status])
  @@map("build_runs")
}

Two separate score columns — trustScore (synthetic, computed from lints + security + test pass-rate) and reviewScore (human or AI reviewer's 0–1 verdict). Both Decimal(4,3) so 0.000–9.999 with millidecimal precision; the dispatcher's auto-advance compares against the per-project policy blob (§4.2).

4.8 TestRun — Playwright + unit-test execution attached to a BuildRun

Source: schema.prisma:205-225

model TestRun {
  id              String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workItemId      String    @map("work_item_id") @db.Uuid
  buildRunId      String?   @map("build_run_id") @db.Uuid
  agentc2RunId    String?   @map("agentc2_run_id")
  status          String    @default("pending")
  testResults     Json      @default("[]") @map("test_results")
  screenshots     Json      @default("[]")
  playwrightTrace String?   @map("playwright_trace")
  failureDetails  String?   @map("failure_details")
  retryCount      Int       @default(0) @map("retry_count")
  durationMs      Int?      @map("duration_ms")
  tokenCost       Decimal?  @map("token_cost") @db.Decimal(10, 4)
  createdAt       DateTime  @default(now()) @map("created_at") @db.Timestamptz(6)
  completedAt     DateTime? @map("completed_at") @db.Timestamptz(6)
  workItem        WorkItem  @relation(fields: [workItemId], references: [id], onDelete: Cascade)
  buildRun        BuildRun? @relation(fields: [buildRunId], references: [id])

  @@index([workItemId])
  @@map("test_runs")
}

buildRunId is nullable so a free-standing regression / smoke run can exist without a build. playwrightTrace is the URL of the uploaded .zip trace artifact (object-storage path).

4.9 TransitionLog — write-only audit trail of every state change

Source: schema.prisma:227-242

model TransitionLog {
  id        String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  entity    String
  entityId  String   @map("entity_id")
  fromState String   @map("from_state")
  event     String
  guard     String?
  toState   String   @map("to_state")
  actions   String[]
  metadata  Json?
  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)

  @@index([entity, entityId])
  @@index([createdAt])
  @@map("transition_logs")
}

Polymorphic on purpose — no FK back to WorkPackage / WorkItem / BuildRun because the log spans every entity that has a state machine. The (entity, entityId) index makes "show me the full transition history for this row" a single fast query; (createdAt) serves the global activity feed in the operator UI.

4.10 DocArtifact — release notes, user guides, internal docs

Source: schema.prisma:244-259

model DocArtifact {
  id            String      @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  workPackageId String      @map("work_package_id") @db.Uuid
  docType       String      @map("doc_type")
  title         String
  content       String      @default("")
  screenshots   Json        @default("[]")
  status        String      @default("draft")
  agentc2RunId  String?     @map("agentc2_run_id")
  createdAt     DateTime    @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt     DateTime    @default(now()) @map("updated_at") @db.Timestamptz(6)
  workPackage   WorkPackage @relation(fields: [workPackageId], references: [id], onDelete: Cascade)

  @@index([workPackageId])
  @@map("doc_artifacts")
}

docType is the discriminator ("release-notes", "user-guide", "runbook", …). Hangs off WorkPackage rather than WorkItem because docs cover the whole feature, not a single ticket.

5 · Migration strategy

Honest observation. packages/database/prisma/migrations/ contains only migration_lock.toml (provider = postgresql). No timestamped migration SQL files are committed at the time of this UIS. The schema is rolled out via prisma db push (dev) and the migration history is intentionally empty until the schema stabilises post-P1.

Migration choreography for this epic is deliberately staged so the schema can iterate quickly before being frozen:

  • Dev rollout: bun run db:push (packages/database/package.json:11) — declarative push of schema.prisma against DATABASE_URL. Used during the 45h build because the model surface was still moving every few hours.
  • Two DATABASE_URLs: DATABASE_URL is the Supabase pooler endpoint Prisma uses at runtime; DATABASE_URL_MIGRATE is the direct connection used only by prisma migrate dev / migrate deploy so the shadow database can be created without pooler interference. See packages/database/package.json:13-18.
  • Generator: bun run db:generate emits the typed client into node_modules/@prisma/client; the workspace package packages/database/src/index.ts re-exports prisma and Prisma types so the rest of the monorepo never imports from @prisma/client directly.
  • Reset path: bun run db:migrate:reset nukes + replays; migrate-fresh.ts at packages/database/src/migrate-fresh.ts is the bespoke local-only escape hatch when the schema is on fire.
  • Seed: bun run db:seed runs packages/database/src/seed.ts; seed-projects.ts backfills the demo Project rows that the operator UI depends on.
  • Sanity scripts: src/check-wp.ts, src/check-wi.ts, src/check-so.ts, src/check-tl-build.ts — quick Prisma reads that smoke-test each model has the right shape after a push.
  • Cutover plan (post-P1): once the schema is frozen, run prisma migrate dev --name initial against the migrate URL to baseline; commit the first SQL migration; flip CI to prisma migrate deploy for prod. Tracked in C1-0002.

8 · Test plan

8.1 Schema-level checks

  • prisma validate passes; prisma format idempotent.
  • db:push against an empty Supabase DB → db:push again must be a no-op (zero proposed changes). Any drift is a P0.
  • db:generate emits a typed client; packages/database/src/index.ts compiles under tsc --noEmit.

8.2 Constraint correctness

  1. Inserting two StageOutput rows with the same (workPackageId, stage, version) must fail with a Postgres uniqueness violation — proves the audit-trail constraint.
  2. Deleting a WorkPackage must cascade to all its StageOutput / Review / WorkItem / DocArtifact rows and (transitively) all BuildRun / TestRun rows. Verify by row count diff.
  3. Deleting a BuildRun must not cascade-delete its TestRun children (FK is optional + no onDelete: Cascade) — test runs can outlive a re-run build.

8.3 Smoke scripts

// packages/database/src/check-wp.ts — sample
const wp = await prisma.workPackage.findFirst({
  include: { stageOutputs: true, reviews: true, workItems: { include: { buildRuns: true, testRuns: true } }, docArtifacts: true }
});
console.log(JSON.stringify(wp, null, 2));

Each of check-wp.ts / check-wi.ts / check-so.ts / check-tl-build.ts must run green on a fresh seeded DB. They are run manually before every push; CI integration is a follow-up.

8.4 End-to-end flow

  1. Hit /work-packages/new, submit the seed form → assert one WorkPackage row (stage = "draft") + one initial TransitionLog row.
  2. Trigger the enrichment stage → assert one StageOutput(stage="enrichment", version=1) + one Review + one TransitionLog.
  3. Continue through every stage on the detail page → final state has rows in WorkItem, BuildRun (with prUrl populated), TestRun, and DocArtifact.
  4. The committed screenshots 01-work-packages-list.png through 10-scope-pending-review-full.png at the repo root are the canonical visual acceptance evidence.

9 · Risks & mitigations

RiskImpactMitigation
Empty migrations/ directory — the schema is rolled via db:push only. There is no committed SQL history; recreating prod from scratch relies on schema.prisma alone. High for prod / DR posture Plan baselines a prisma migrate dev --name initial once the schema is frozen (tracked under C1-0002). Until then, the staging DB is treated as authoritative for any schema-shape question; nightly pg_dump snapshot to S3 covers DR.
Open-vocabulary stage / status strings — every state column is a free String, not a Prisma enum. Typos at the call-site silently land in the DB and the operator UI can show garbage states. Med The xstate machine in apps/frontend/src/lib/actions/router/ is the single writer for state columns. A Prisma client extension (planned in C1-0002) validates stage / status against the machine's known states on every create / update. The free-string choice is deliberate during P1 so new stages can be added without a migration round-trip.
Polymorphic TransitionLog.entity with no DB-level FK — orphaned log rows can outlive their entity, and entity renames silently break the audit replay. Med The entity values come from a single const in the router types module (apps/frontend/src/lib/actions/router/types.ts); a unit test fails CI if a router writes an unknown entity. (entity, entityId) index makes orphan detection a cheap nightly scan; orphans are kept as evidence rather than pruned.
v2 table-name suffix — work_packages_v2 is a smell that costs cycles every time someone joins it manually. Low Schedule the legacy work_packages drop + rename to land in the first real SQL migration. Until then, a SQL view work_packages AS SELECT * FROM work_packages_v2 covers any ad-hoc readers without breaking them.
JSON column proliferation — scope / acceptanceCriteria / techSpec / reviewNotes / securityFindings / testResults / screenshots / autoAdvancePolicy are all opaque JSON. Query-by-shape requires Postgres JSONB operators; refactoring later is expensive. Med Zod schemas live next to each writer and are enforced at the action boundary so the runtime shape is contractual even when the DB type isn't. A periodic schema-extract job (post-P1) flattens hot fields into typed columns when query patterns stabilise.
