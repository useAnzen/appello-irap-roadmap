1 · Objective

Sections 9 ("End-to-End Test Plan") and 10 ("Success Criteria") of agentc2/docs/sdlc-e2e-analysis-and-test-plan.md are today's de-facto requirements doc for the SDLC pipeline — but they are written as engineering test recipes, not as PM-style user stories with priority and acceptance. This epic rewrites the same content into formal UIS-NN user stories with ranked acceptance criteria covering C1 (SDLC Flywheel) and C3 (ATLAS Knowledge Base), feeding the UIS Generator with a clean, prioritised backlog.

2 · Acceptance Criteria

  • Every test case in §9 maps to ≥1 user story in the rewritten doc (no gaps).
  • Each story uses Connextra format: As a <persona>, I want <capability> so that <outcome>; persona drawn from {SDLC Operator, Reviewer, Customer Success, Field Lead}.
  • Each story has 3–10 acceptance criteria written as Given/When/Then, testable without ambiguity.
  • Stories are ranked P0/P1/P2 with explicit rationale (MoSCoW or RICE-lite).
  • C1 and C3 sections are separated; cross-component dependencies are explicit ("depends on C3-0231").
  • Doc lives at agentc2/docs/product/sdlc-requirements.md; sdlc-e2e-analysis-and-test-plan.md updated to link to it as the source-of-truth for requirements.
  • Backlog rows exported as JSON (requirements.json) consumable by the UIS Generator agent.

3 · Architecture & Approach

The translation is mechanical-but-judgmental: each E2E case becomes a story; each assertion becomes one AC. Priority assignment is the new work — done against a simple rubric:

  • P0 — required to ship 1.33/1.34 release or unblocks the autonomous pipeline.
  • P1 — quality/observability that the customer touches.
  • P2 — nice-to-have, internal tooling.

JSON schema for backlog

type Story = {
  id: string;                  // UIS-NN-SM
  component: 'C1' | 'C3';
  persona: string;
  capability: string;
  outcome: string;
  ac: { gwt: string; testable: boolean }[];
  priority: 'P0' | 'P1' | 'P2';
  rationale: string;
  dependsOn: string[];         // story ids
  sourceTestCases: string[];   // §9 references
};

4 · Implementation Phases (30h)

  1. Section parser — pull §9 + §10 into structured rows (4h).
  2. Story drafting — Connextra rewrite for C1 (10h).
  3. Story drafting — Connextra rewrite for C3 (8h).
  4. Priority ranking workshop + rationale (4h).
  5. JSON export + UIS-Generator consumption test (4h).

5 · Risks & Mitigations

  • Translation loses test-level specificity. Mitigation: every story carries sourceTestCases[] back-reference.
  • Personas don't match real users. Mitigation: 1-hour validation interview with each persona before sign-off.
  • Drift between requirements doc and test plan. Mitigation: single CI check that asserts every §9 case is covered by ≥1 story id.
