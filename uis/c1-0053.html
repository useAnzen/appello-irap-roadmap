1 · Objective — close the AI-authored PR loop

Make the end-to-end loop from AI-authored change to production deploy fully automated and gated, with no manual steps in the green path and explicit hard blocks in the red path:

  1. Open the PR — a cursor-* agent (or any sdlc-* agent) opens a pull request using the native GitHub tool surface (github_create_pull_request via MCP, or the binary-safe github-push-files for changes that include screenshots / SBOMs). The PR title must carry the Jira key as a prefix.
  2. Automated review — sdlc-reviewer reads the diff, computes a per-PR trust score from the CI signal set, and posts the score as a PR comment (or check run).
  3. CI gate — .github/workflows/security-gates.yml runs on every pull_request: type-check, lint, unit + integration tests, dependency audit, SCA scan, license compliance, SBOM, and secret scan.
  4. Merge decision — verify-* / wait-for-checks tools block auto-merge below the trust threshold; only PRs at-or-above threshold flip from draft → ready and merge via the platform's merge-pull-request tool.
  5. Deploy trigger — push to main fires .github/workflows/deploy-do.yml which builds Docker images on the Digital Ocean droplet, restarts containers via Docker Compose, health-checks agentc2.ai, and reports back to Slack tagged with the Jira key from the PR title.

The result is a single fire-and-forget path: Cursor agent → PR → CI → trust gate → auto-merge → deploy, with Jira tracking every step.

2 · Acceptance Criteria

  1. A cursor-* (or other sdlc-*) agent can open a PR end-to-end using only the consolidated native GitHub tool surface — no human keystrokes between "agent commits" and "PR exists with Jira key in title."
  2. Every PR triggers .github/workflows/security-gates.yml automatically (on: pull_request), and the workflow runs the full bundle: type-check, lint, unit + integration tests, bun audit, audit-ci --high, license compliance, CycloneDX SBOM, and Gitleaks secret scan.
  3. sdlc-reviewer computes a trust score per PR from the CI signal set and posts it back to the PR (comment or check run) before the merge decision is taken.
  4. Auto-merge is hard-blocked below the trust threshold: verify-* / wait-for-checks refuse to call merge-pull-request until both (a) all required checks are green and (b) trust score ≥ threshold.
  5. PR title format <JIRA-KEY> <title> is enforced; the Jira key is parsed and threaded through to the Slack deploy notification.
  6. A merge to main deterministically triggers .github/workflows/deploy-do.yml; the deploy job SSHes to the droplet, builds with docker compose -f docker-compose.production.yml build, brings the new containers up, and health-checks https://agentc2.ai/, /login, and /admin.
  7. The deploy notifies Slack on success and failure, carrying ${{ github.sha }} and the first line of the commit message (which embeds the Jira key).
  8. Native GitHub tool IDs across the platform are consolidated and consistent: github_create_pull_request and github_add_issue_comment are the canonical IDs (the hyphenated github-* variants have been migrated by scripts/consolidate-github-tools.ts), so agents and workflows speak one vocabulary.
  9. Binary artifacts (screenshots, SBOMs, PDF receipts) push correctly via github-push-files with encoding: "base64" per the fix in docs/sdlc/09-github-binary-encoding-response.md.
  10. The whole loop is observable: failed CI, missing trust score, threshold-block, and deploy failure each surface a distinct Slack signal pointing at the GitHub Actions run URL.

4 · Trust score algorithm — what sdlc-reviewer computes

The trust score is the gate that decides whether a PR is allowed to auto-merge. It is computed per PR by sdlc-reviewer from the signal set produced by .github/workflows/security-gates.yml plus the diff itself. The inputs are deterministic; the score is just a weighted sum of signals that already exist in the workflow and the GitHub API.

SignalSourceWeightPass condition
Type checkbunx turbo type-checkrequiredexit 0
Lintbunx turbo lintrequiredexit 0
Unit + integration testsbun run testrequiredexit 0
Dependency auditbun auditadvisorylogged, non-blocking (|| true)
SCA scanbunx audit-ci --highadvisoryno high severity
License compliancebunx license-checkeradvisoryno GPL family in --production graph
SBOM generation@cyclonedx/cyclonedx-npmrequired-artifactuploaded as workflow artifact sbom, 90d retention
Secret scangitleaks/gitleaks-action@v2requiredno findings in diff
Tool-ID hygieneconsolidate-github-tools.ts (dry-run)requiredzero pending replacements
PR title formatregex ^[A-Z]+-\d+ requiredJira key prefix present

The score formula is the simplest thing that works: any required failure → score = 0 → auto-merge blocked. Advisory signals reduce the score but do not block, so a noisy bun audit alone cannot freeze the pipeline. The threshold for auto-merge is the score that means "every required check passed and no more than one advisory failed" — encoded as a numeric threshold so the policy can be tuned without redeploying the agent.

Why this is anchored to the workflow, not to the model. The signal set above is exactly what .github/workflows/security-gates.yml already runs on every pull_request. sdlc-reviewer reads the check run results via the GitHub API rather than re-running anything — the trust score is a deterministic projection of the CI run, not an LLM opinion about the diff. That keeps the gate auditable and reproducible.
Binary-encoding hardening (cross-ref). Trust-score collection includes any github-push-files artifact uploads. Per docs/sdlc/09-github-binary-encoding-response.md, agents pushing screenshots or PDFs must use encoding: "base64"; the score docks any PR that uploaded a binary with the wrong encoding (detected by GitHub's content-type rather than relying on the agent's claim).

7.3 How the walker rewrites every tool reference

Source: scripts/consolidate-github-tools.ts:67-107

function walkAndReplace(
    obj: JsonValue,
    path: string = ""
): { result: JsonValue; changed: boolean } {
    if (obj === null || obj === undefined) return { result: obj, changed: false };

    if (typeof obj === "string") {
        const replacement = replaceToolId(obj);
        if (replacement) return { result: replacement, changed: true };
        return { result: obj, changed: false };
    }

    if (Array.isArray(obj)) {
        let anyChanged = false;
        const newArr = obj.map((item, i) => {
            const { result, changed } = walkAndReplace(item, `${path}[${i}]`);
            if (changed) anyChanged = true;
            return result;
        });
        return { result: newArr, changed: anyChanged };
    }

    let anyChanged = false;
    const newObj: Record<string, JsonValue> = {};
    for (const [key, val] of Object.entries(obj)) {
        if (key === "toolId" && typeof val === "string") {
            const replacement = replaceToolId(val);
            if (replacement) {
                newObj[key] = replacement;
                anyChanged = true;
                continue;
            }
        }
        const { result, changed } = walkAndReplace(val as JsonValue, `${path}.${key}`);
        newObj[key] = result;
        if (changed) anyChanged = true;
    }
    return { result: newObj, changed: anyChanged };
}

7.4 Direct-column rewrite — AgentTool

Source: scripts/consolidate-github-tools.ts:109-134

async function migrateDirectColumns() {
    console.log("Scanning AgentTool...");
    const agentTools = await prisma.agentTool.findMany({
        where: { toolId: { in: Object.keys(REPLACEMENT_MAP) } },
        select: { id: true, agentId: true, toolId: true, agent: { select: { slug: true } } }
    });
    for (const row of agentTools) {
        const newId = REPLACEMENT_MAP[row.toolId]!;
        changes.push({
            table: "AgentTool", id: row.id, field: "toolId",
            detail: `agent:${row.agent.slug} | ${row.toolId} → ${newId}`
        });
        if (APPLY) {
            const existing = await prisma.agentTool.findFirst({
                where: { agentId: row.agentId, toolId: newId }
            });
            if (existing) {
                await prisma.agentTool.delete({ where: { id: row.id } });
            } else {
                await prisma.agentTool.update({ where: { id: row.id }, data: { toolId: newId } });
            }
        }
    }
}

8 · Jira integration — how the PR title drives the deploy tag

There is no Jira webhook in this loop. Jira tracking rides on the PR title contract and the squash-merge commit message — the lowest-overhead way to thread a key through the entire pipeline.

8.1 Title contract

Every PR opened by an sdlc-* agent must have title <JIRA-KEY> <short description>, e.g. SDLC-123 Wire trust-score threshold into auto-merge gate. The regex enforced by sdlc-reviewer is ^[A-Z]+-\d+ \S; PRs without a key score zero on that signal and are blocked from auto-merge (see §4).

8.2 Why squash-merge matters here

The merge-pull-request tool squashes by default. GitHub uses the PR title as the squash commit's first-line message. That means the commit hitting main looks like:

SDLC-123 Wire trust-score threshold into auto-merge gate (#1742)

which is exactly what .github/workflows/deploy-do.yml reads in the Slack-notify step:

COMMIT_MSG=$(echo '${{ github.event.head_commit.message }}' | head -1 | sed 's/"/\\"/g')

So the Jira key flows automatically from the PR title → squash commit → head -1 → Slack message. The deploy notification carries the key without any extra integration.

10 · Risks & mitigations · evidence · cross-references

10.1 Risks & mitigations

RiskImpactMitigation
False-positive trust failures — an advisory signal (e.g. bun audit) spikes and drags the score below threshold, freezing the pipeline for a legitimate change. Auto-merge stalls; humans have to bypass. All advisory signals are wrapped in || true in the workflow, so they emit signal without failing CI. The score formula caps the penalty from any single advisory at less than the threshold delta — one noisy advisory cannot block on its own. sdlc-reviewer also includes a "this would have merged at last week's dependency snapshot" note when an advisory regression looks transient.
GitHub API rate limits — sdlc-reviewer and the verify-* tools poll check-run state per PR; under burst load the org-level limit can throttle the loop. Trust score posts late; verify-* sees stale state and either over- or under-blocks. All native GitHub tool calls go through the shared resolveGitHubToken() path and respect ETag / conditional-request semantics. Polling cadence is exponential-backoff capped at 60s; sdlc-reviewer subscribes to check_suite events through GitHub's webhook surface where possible instead of polling. The org-level budget is monitored and surfaced as its own advisory signal so a near-exhaustion shows up in the score before it causes a hard failure.
Jira → deploy race — a human edits the PR title after CI starts but before merge, breaking the Jira key contract and leaving the deploy Slack message keyless. Deploy notification cannot be traced back to a Jira ticket; auditability gap. The title regex is re-checked at merge time (not just at PR-open time); verify-* refuses to call merge-pull-request if the title no longer matches ^[A-Z]+-\d+ \S. Because merge-pull-request squashes by default, the committed first line is whatever the title is at merge — keeping head -1 in the deploy workflow as the single source of truth for Slack.
Binary-encoding regression — a future agent change forgets to pass encoding: "base64" and pushes corrupt screenshots into a PR.
