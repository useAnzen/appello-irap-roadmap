PA-1 — Planning Agent
Sprints 5–6 · Jul 2026 ~4 weeks Depends: BRA-1 (R1) Team: Corey, NH1, NH2
Takes a Jira ticket and the Appello codebase AST, produces an ordered subtask list with target file paths, acceptance criteria, and estimated complexity. This is the “Planner” micro-agent from the Flywheel 2.0 multi-agent architecture.
Implementation Plan
  • Week 1: Ticket ingestion & codebase parsing — Build the Jira ticket reader that extracts summary, description, acceptance criteria, and linked issues via Jira API. Parse Appello monorepo structure (TypeScript/React/Express/Prisma) into an AST representation the agent can reason over.
  • Week 1: Context retrieval from ATLAS — Query the Supabase vector store (1000-token chunks, 1536-dim embeddings) for related meeting transcripts, prior tickets, and architecture decisions relevant to the ticket.
  • Week 2: Subtask decomposition engine — Implement the master plan → sub-plan pattern (from Corey’s Flywheel 2.0 workflow): ticket → ordered subtask list with target file paths, change description, and complexity estimate per subtask.
  • Week 3: Plan audit step — Build the self-audit layer: after generating a plan, the agent reviews its own output for completeness, circular dependencies, and missing edge cases. Flags items for human review when confidence is below threshold.
  • Week 4: Validation against real tickets — Run against 15 real construction-domain Jira tickets (e.g., timesheet bugs, financial summary errors, equipment form issues). Compare AI-generated plans to the actual developer-written plans for those tickets. Score for completeness and accuracy.
Acceptance Criteria
  • Reads any Q21030 Jira ticket via API and extracts structured fields
  • Produces ordered subtask list with file paths referencing real Appello monorepo files
  • Each subtask includes: target file(s), change description, acceptance criteria, complexity (S/M/L)
  • Self-audit step catches ≥80% of missing edge cases (measured against human review)
  • Validated against 15 real tickets with ≥70% plan accuracy vs. human baseline
  • Execution time <120 seconds per ticket; cost <$2.00 per plan (model routing: Haiku for parsing, Sonnet for reasoning)
Architecture
Runs within the Catalyst/Agent C2 orchestration platform with 15 connected tools (Jira, GitHub, Slack, Supabase). Uses model auto-routing: Claude 3 Haiku for ticket parsing and simple lookups, GPT-4o/Claude Sonnet for complex reasoning and plan generation. Context window managed by chunking the Appello monorepo into per-module summaries.
Data Flow
Jira Ticket → Ticket Parser → ATLAS Context Retrieval → Codebase AST Loader → Plan Generator (LLM) → Self-Audit (LLM) → Subtask JSON Output
Risks & Mitigations
  • Appello monorepo is large (~resource-intensive to parse). Mitigation: Pre-index module boundaries; only load relevant modules per ticket.
  • Plan quality degrades for cross-cutting tickets. Mitigation: Self-audit step + human-in-the-loop checkpoint for L-complexity plans.

C1 · SDLC Flywheel · Agent
DA-1 — Development Agent v1
Sprints 6–7 · Jul–Aug 2026 ~4 weeks Depends: PA-1 Team: Corey, NH1, NH2
Takes a Planning Agent subtask, generates TypeScript/React code changes targeting the correct files in the Appello monorepo. Outputs as a PR-ready diff. This is the “Builder” micro-agent — it writes code, runs type checks, writes tests, ensures tests pass, runs the formatter, and creates a formatted PR.
Implementation Plan
  • Week 1: Subtask → code generation pipeline — Accept PA-1 subtask JSON, load target file(s) from Appello monorepo, generate code diff using LLM with full file context. Handle TypeScript, React components, Prisma schema changes, and GraphQL resolver updates.
  • Week 1: Appello coding conventions — Train on Appello patterns: Turbo Repo structure, PNPM workspace layout, Next.js 13 app router conventions, Tailwind CSS utility classes, Apollo Server GraphQL patterns. Embed conventions as system prompt context.
  • Week 2: Build → validate loop — After generating code: run tsc --noEmit (TypeScript compiler), eslint, prettier --check. If failures, feed errors back to LLM for self-correction (max 3 retry cycles).
  • Week 3: Test generation — Auto-generate Playwright tests for UI changes, Jest unit tests for API/service changes. Run generated tests against the code diff. If tests fail, iterate on both code and tests.
  • Week 4: PR formatting & output — Generate clean git diff, create GitHub PR with structured description (linked Jira ticket, change summary, test results, files modified). Apply Appello’s PR conventions: branch name = ticket number, PR title = ticket summary.
Acceptance Criteria
  • Generates syntactically valid TypeScript/React code targeting correct Appello monorepo files
  • Self-correction loop resolves ≥90% of tsc and ESLint errors within 3 retries
  • Auto-generates at least 1 relevant test per subtask (Playwright for UI, Jest for API)
  • Creates well-formatted GitHub PR following Appello conventions
  • End-to-end time from subtask to PR <15 minutes; cost <$8.00 per subtask
  • Human code review required before merge (human-in-the-loop checkpoint)
Architecture
Builds on the Catalyst platform’s GitHub integration. The agent operates in a sandboxed git worktree per subtask, preventing cross-contamination between concurrent tasks. Uses the same model routing as PA-1 but biases toward higher-capability models (GPT-4o / Claude Sonnet) for code generation, with Haiku handling the type-check/lint/format feedback loop.
Data Flow
PA-1 Subtask → File Loader → Code Generator (LLM) → tsc + ESLint + Prettier → Self-Correct Loop → Test Generator → Test Runner → PR Creator (GitHub API)
Risks & Mitigations
  • Generated code may compile but be semantically wrong. Mitigation: Human-in-the-loop review is mandatory; Validation Agent (VA-1) adds automated semantic checks.
  • Cost per subtask may exceed budget with large file contexts. Mitigation: Model routing + context windowing; only load files within 2 hops of the target.
  • Retry loops can be expensive. Mitigation: Cap at 3 retries; escalate to human with error context if unresolved.

VA-1 — Validation Agent
Sprints 7–8 · Aug 2026 ~4 weeks Depends: DA-1 Team: Corey, NH1, Emma Mann
Receives AI-generated PRs, runs static analysis + existing Playwright test suites + Appello-specific coding pattern checks. Returns a quality score and blocks merge if below threshold. This is the “Auditor” and “Tester” micro-agents combined.
Implementation Plan
  • Week 1: Static analysis pipeline — Run tsc --noEmit, ESLint with Appello config, Prettier check on every AI-generated PR. Aggregate results into structured report. Integrate with existing GitHub Actions CI.
  • Week 1: Appello pattern checks — Custom lint rules for Appello-specific patterns: Prisma schema conventions, GraphQL resolver structure, React component patterns (hooks usage, Tailwind classes), Next.js app router conventions.
  • Week 2: Existing test suite execution — Run the existing Playwright test suite against the PR branch in a Docker Compose environment. Parse results, identify regressions vs. pre-existing failures.
  • Week 3: Quality scoring model — Composite score from: type safety (0-25), lint cleanliness (0-25), test pass rate (0-25), pattern compliance (0-25). Configurable threshold (default: 70/100) for merge gate. Score card stored for agent self-improvement (Flywheel feedback loop).
  • Week 4: Merge gate & feedback — GitHub status check integration: block merge if below threshold. Generate structured feedback for DA-1 to retry. Human override available for edge cases. Dashboard for tracking quality trends over time.
Acceptance Criteria
  • Runs full static analysis pipeline on every AI-generated PR in <5 minutes
  • Executes existing Playwright suite and correctly identifies regressions (not pre-existing failures)
  • Produces composite quality score (0-100) with per-category breakdown
  • Blocks GitHub merge for PRs below configurable threshold
  • Generates actionable feedback that DA-1 can consume for self-correction
  • Score cards stored in pipeline tracking schema (SDLC-DB) for Flywheel self-improvement
Architecture
Integrates with GitHub via webhooks (PR opened/updated events). Spins up a Docker Compose test environment per PR (reusing Travis’s existing CI infrastructure). Leverages existing AI code review tools already in use at Appello: Cursor BugBot and Claude for business logic review, plus the new quality scoring model for automated gating.
Data Flow
GitHub PR Webhook → Checkout Branch → Static Analysis (tsc + ESLint + Prettier) → Docker Test Env → Playwright Suite → Quality Score Calculator → GitHub Status Check + Score Card → SDLC-DB

E2E-1 — End-to-End Pipeline
Sprints 8–10 · Aug–Sep 2026 ~6 weeks Depends: VA-1 Team: Corey, Chris, Travis, NH1, NH2
Wire the full pipeline: Jira ticket → Bug Report Agent → Planning Agent → Development Agent → Validation Agent → PR. Stage transition logic, status tracking, error handling, and the Flywheel feedback loop (score cards drive agent self-improvement).
Implementation Plan
  • Weeks 1–2: Orchestrator agent — Build the orchestrator that manages the pipeline sequence: BRA-1 → PA-1 → DA-1 → VA-1. Implements the Flywheel patterns: heartbeat (cron trigger), scarcity (limited tasks per cycle), stage transition logic. Runs within Catalyst/Agent C2 with all 15 connected tools.
  • Week 2: Error handling & retry strategies — Five retry modes: immediate retry, exponential backoff, skip-and-continue, escalate-to-human, abort-pipeline. Each agent failure type maps to a specific retry strategy. Dead letter queue for unrecoverable failures.
  • Weeks 3–4: Status tracking & observability — Each pipeline run tracked in SDLC-DB (8 tables from R1). Real-time status: queued → planning → developing → validating → awaiting-review → merged/rejected. Structured logging with cost tracking per stage.
  • Week 4: Human-in-the-loop checkpoints — Two mandatory checkpoints: (1) after Planning Agent produces the plan, (2) after Validation Agent scores the PR. Slack notifications to the assigned reviewer. Configurable auto-approve for high-confidence, low-complexity tickets.
  • Weeks 5–6: Flywheel feedback loop — Score cards from VA-1 feed back into agent improvement. Track: plan accuracy over time, code quality trends, cost per ticket, time per stage. Self-healing: agents can create tickets for themselves when they identify patterns of failure.
  • Week 6: Load testing & cost optimization — Run 50 tickets through the full pipeline. Measure throughput, identify bottlenecks, optimize model routing for cost. Target: 10 tickets/day throughput at <$15/ticket average.
Acceptance Criteria
  • Full pipeline runs autonomously from Jira ticket to GitHub PR with two human checkpoints
  • All five error handling strategies implemented and tested
  • Pipeline status visible in real-time via SDLC-DB queries
  • Cost tracking per stage and per ticket with monthly reporting
  • Flywheel feedback loop: quality scores trend upward over 50-ticket run
  • Throughput: ≥10 tickets/day; average cost <$15/ticket
  • Slack notifications for human checkpoints delivered within 30 seconds
Architecture
The orchestrator is the central coordinator of the Flywheel 2.0 multi-agent system. It manages a swarm of specialized micro-agents (Planner, Builder, Auditor/Tester, Bug Fixer) with the human developer as “commander.” Built on the Catalyst platform with Jira, GitHub, Slack, and Supabase integrations. Pipeline state stored in the SDLC-DB schema deployed in R1.
Full Pipeline Flow
Jira Ticket → BRA-1 (Bug Report) → PA-1 (Plan) → Human Checkpoint 1 → DA-1 (Code) → VA-1 (Validate) → Human Checkpoint 2 → GitHub PR Merge → Score Card → Flywheel

ATLAS-2 — MySQL Schema Vectorization
Sprints 6–9 · Jul–Sep 2026 ~8 weeks Depends: ATLAS-1 (R1) Team: Chris
Vectorize Appello’s ~200-table MySQL schema into ATLAS. Every table definition, column, relationship, and business logic annotation becomes queryable via semantic search — enabling NL→SQL in the next phase.
Implementation Plan
  • Weeks 1–2: Schema extraction pipeline — Read Appello’s schema.prisma file (source of truth, located at apps/api/prisma). Extract all models, fields, types, relations, and enums. Generate human-readable descriptions of each table’s purpose and relationships.
  • Weeks 2–3: Business logic annotation — Enrich schema with business context from ATLAS meeting transcripts and internal docs. Example: the TradeLevelRate model isn’t just “a rate table” — it’s the billable rate for a specific trade level in an employee agreement, used for labor invoicing.
  • Weeks 3–4: Vectorization into Supabase — Chunk schema descriptions into 1000-token segments (matching existing ATLAS format). Generate 1536-dimension embeddings via OpenAI embedding model. Store in Supabase pgvector alongside existing meeting/document vectors.
  • Weeks 5–6: Cross-reference layer — Link schema vectors to related meeting transcripts and Jira tickets already in ATLAS. When querying about a table, retrieval also surfaces relevant conversations about that feature.
  • Weeks 7–8: Validation & query testing — Test semantic search accuracy: “which table stores employee billable rates?” should return TradeLevelRate. Validate against 100 natural language questions about Appello’s data model.
Acceptance Criteria
  • All ~200 Prisma models vectorized with human-readable descriptions
  • Business logic annotations for top 50 most-queried tables (based on customer ticket frequency)
  • Semantic search returns correct table for ≥85% of natural language queries
  • Cross-references link schema vectors to relevant meeting transcripts and tickets
  • Vector store query latency <3 seconds (matching existing ATLAS performance)
  • Incremental update pipeline: new Prisma migrations auto-vectorize within 24 hours
Architecture
Extends the existing ATLAS infrastructure (Supabase Postgres with pgvector, 729+ transcripts already vectorized). Schema vectors live in the same vector store but with a source_type = 'schema' tag for filtered retrieval. Uses the same 1000-token / 1536-dimension format as meeting transcripts, ensuring cross-referencing works seamlessly.

C3 · ATLAS Knowledge Base · ML
NL2SQL-1 — Natural Language → SQL Prototype
Sprints 9–10 · Sep 2026 ~4 weeks Depends: ATLAS-2 Team: Chris, Nathan
Natural language queries against Appello production data. “Show me all jobs over budget this month” → correct SQL → formatted result. This is the foundation for the customer-facing AI toolboxes in R4.
Implementation Plan
  • Week 1: Query intent classification — Classify incoming natural language into query type: financial (jobs, budgets, costs), labor (timesheets, crews, hours), scheduling (dates, deadlines), equipment (usage, maintenance), materials (consumption, POs). Route to appropriate schema context from ATLAS-2.
  • Week 1: SQL generation pipeline — Use ATLAS-2 schema vectors to identify relevant tables/columns. Feed table context + user question to LLM. Output: syntactically valid MySQL query. Validate against Appello’s schema before execution.
  • Week 2: Safety & guardrails — Read-only query enforcement (SELECT only, no mutations). Per-tenant data isolation (WHERE clause injection for customer ID). Query timeout limits. Result set size caps. Sensitive column masking (e.g., salary, SIN).
  • Week 3: Result formatting — Transform raw SQL results into human-readable responses. Tables for list data, summaries for aggregations, charts descriptions for trends. Construction-specific formatting (e.g., currency for costs, hours for labor).
  • Week 4: Validation with construction queries — Nathan provides 50 questions a construction PM, foreman, or bookkeeper would actually ask. Test each: NL → SQL → result → human verification. Target: ≥80% correct answers on first attempt.
Acceptance Criteria
  • Correctly generates MySQL queries for ≥80% of 50 construction-specific test questions
  • All queries are read-only with per-tenant data isolation enforced
  • Query execution time <10 seconds including LLM processing
  • Results formatted in human-readable language appropriate to construction domain
  • Graceful handling of ambiguous queries (asks for clarification rather than guessing)
  • Cost per query <$0.50 (model routing: Haiku for classification, Sonnet for SQL generation)
Architecture
Two-stage LLM pipeline: (1) query ATLAS-2 vectors to identify relevant schema context, (2) feed schema context + user question to code-generation model for SQL output. Queries execute against a read-replica of the customer’s MySQL database (DigitalOcean isolated container). Results never cross tenant boundaries.
